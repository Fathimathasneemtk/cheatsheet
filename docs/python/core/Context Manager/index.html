Context Manager in Python

  • Python's context mangers provide nice way to automatically set up and clean up resources, ensuring they are properly managed even aftetr error occurs.

Why do we need Context Manager ?​

  • Automatic cleanup : Frees resources(close file) like files or db connections without manual effort.eg : GL
  • Avoid resource leaks : Ensure proper release(close file) even if error occur.
  • Cleaner code : Replaces verbos like try-finally blocks with with.
  • Exception safe : Always perform cleanup, even on exceptions.
  • Custom control : Lets you manage any resource using __enter__ and __exit__.

Risk of Not Closing Resource​

  • Failing to close resources like files will leads to serious problems like exhausting available system file disrcriptors.This will cause the program or even the entire system to crash .
  • Discriptor - Claim ticket or token the Os gives to programs when the program asks access to OS for the interaction of file, db, network connection.
file_descriptors = []
for x in range(100000):
    file_descriptors.append(open('test.txt', 'w'))

This will raise

Traceback (most recent call last):
  File "context.py", line 3, in 
OSError: [Errno 24] Too many open files: 'test.txt'

This happens bcz of somany files are left open without being closed.

Built-in Context Manger for File Handling​

The with statement provide builtin context manger that ensures file is automatically closed once you are done with it, even if error occurs.

with open("file1.txt") as f:
    data=f.read()

Here we dont wants excplicitly call f.close() and protects aganist resource leakage.

Creating Custom Context Manager Class​

A Class based context manger has 2 methods,

* __enter__() : sets up the resource and returns it

  • __exit__() : cleanup the resources eg:closes file
  • Example: Shows how Python initializes the object , enters context and runs the block and then exits while cleaning up.

Class ContextManger:
    def __init__(self):
        print("Iniit method called")

    def __enter__(self):
        print("enter menthod called")
        return self

    def __exit__(self,exc_type,exc_value,exc_traceback):
        print("exit method called")

with ContextManager() as manager:
    print("with statement block")

output

init method called
enter method called
with statement block
exit method called

Explanation

  • __init__ : initailizes object
  • __enter__: runs at the start of the with block and returns the object.
  • The block executes (print statement)
  • __exit__ : This is called afetr the blocks ends to handle cleanup

A.File Management Using Context Manager​

Here a FileManager class to handling opening, reading/writing and automatically closing file.

Example


class FileManager:
    def __init__(self,mode,filename):
        self.filename=filename
        self.mode=mode
        self.file=None

    def __enter__(self):
        self.file=open(self.filename,self.mode)
        return self.file

    def __exit__(self,exc_type,exc_value,exc_traceback):
        self.file.close


with FileManager('test.txt','w') as f:
    f.write("Test")

print(f.closed)

output

True

Explanation

  • __enter__() opens the file and retruns it
  • Inside the with block , use the file object
  • __exit__() ensures the file closed automatically.
  • print(f.closed) confirms the file is closed.

Database Connection Management with Context Manager​

Here is a simple database connection management sysytem. The number of db connections opens at a time is limited. So we need context managers to help this out.


from pymongo import MongoClient

class MongoDbConnectionManager:
    def __init__(self,hostname,port):
        self.hostname=hostname
        self.port=port
        self.connection=None

    def __enter__(self):
        self.connection=MongoClient(self.hostname, self.port)
        return self.connection

    def __exit__(self,exc_type,exc_value,exc_traceback):
        self.connection.close()

with MongoDBConnectionManager('localhost',5432) as mongo:
    collection=mongo.SampleData.test
    data= collection.find_one({'__id':1})
    print(data.get('name'))

Explanation

  • __enter__() opens MongoDB connection
  • mongo inside with block is the client(here mongo) object
  • You can perform database operations
  • __exit__() closes the connection automatically.

Internal Statement of with​

For an object to work with with , it must implement two special methods(often called "dunder" or double-underscore methods):__enter__ and __exit__.This pair is called as Context Manager Protocol.

# What you write

with open("data.txt","w") as f:
    f.write("Hello !")

#Roughly what python executes internally:
manager=open("data.txt","w")
f=manager.__enter__()
try:
    f.write("hello")
finally:
    manager.__exit__(None,None,None)

Life cycle of context manager:

  1. Evaluation: Python evaluates the expression next to with to get the context manager object.
  2. Setup(__enter__()):The manager's __enter__() method is called. Whatever this method returns is assigned to the variable after the as keyword(if present).NOTE : The manager doesn't have to return itself; it can return anything.
  3. Execution: The code block inside the with statement runs.
  4. Cleanup(exit()): When control leaves the block (Whether smoothly or due to an error), the __exit__ method is triggered.

Handling Exception in __exit__​

The exit method takes 3 arguments exc_type,exc_value and exc_tb.

  • If the block ran with no errors , all the three parameters as None.
  • If an error occured , Python passes the exception's class , the exception instance, and the traceback.

The Return Value Rule : if __exit__ returns True, it swallows the exception(exception catched but swallows), and excecution continues normally after the with block.If returns False or None, Python lets the exception propagate upward to be handled elsewhere.

  • Think of __exit__ as having two distinct responsibilities: a. The Cleanup Phase : This is where we actually close the file,release lock,or disconnect from db.This happens no matter what.This happens in the body of the __exit__() before return. b. The return Value: This is where you decide what to do if an error happened during execution.

Simplifying with contextlib.contextmanager​

  • Writing an entire class with __enter__ and __exit__ methods can feel boilerplate-heavy for simple tasks.Python provides a shortcut decorator in the standard library called @contextlib.contextmanager.
  • This decorator turns a plain generator function into a fully functioning context manager.
  • Instead of splitting your logic into 2 methods, write it allin one place using try...finally block and a yield statement.
from contextlib import contextmanager

@contextmanager
def database_connection(db_name):
    #Everyting before 'yield' is the __enter__ phase ---
    print(f"Connecting to {db_name}")
    try:
        yield f"Connection object for {db_name}" # This is what 'as' variable gets (returns of the __enter__ phase)
    # Everything after 'yield' (or in finally) is the __exit__ phase ---
    except Exception as e:
        print(f"An error occured")
        raise # Make sure to reraise if we dont want to swallow the error (Forces __exit__() to return False or None--The excption propagates and crahses (may be in outer 'try--except'))
        # if there is no "raise" then forces __exit__() to return True- swallows exception
    finally:
        print("Closing the db connection safely")

with database_connection("AnalyticsDb")as db:
    print(f"Running queries on :{db}")

How the Generator Transition works​

  1. When the with block starts , the function runs up until hit yield
  2. The function pauses at yield, and the yielded value is passed to your as variable.
  3. Your body code inside with block runs.
  4. Once your code finishes (or raises an error),control is handled back to the generator where it left off, forcing the except or finally or both blocks to execute.
