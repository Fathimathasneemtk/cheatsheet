## 1. What Is the WebSocket API?

**WebSocket API** is a browser-side JavaScript API used to open a **persistent connection** between a client (e.g., a browser) and a server, so both sides can send data to each other without repeatedly making new HTTP requests.
- It is **bidirectional** and **full-duplex** — data can move both ways at the same time, over a **single connection**.It enables true **real time communication**.
- Once the connection is established, the server can push new data to the browser immediately, and the browser can send data back whenever it wants.

> **Important distinction:** The **WebSocket API** is not the same thing as the **WebSocket protocol**.
> - The **protocol** defines *how* client and server communicate at the network level.
> - The **API** is the JavaScript interface the browser provides for using that protocol (`WebSocket`, `onopen`, `onmessage`, etc.).

### Difference between websocket and webRTC**
* WebSocket is built for Client-to-Server communication (like a browser talking to a backend database).

* WebRTC (Web Real-Time Communication) is built for Peer-to-Peer (P2P) communication (like a browser talking directly to another browser).
* WebRTC is not a single protocol — it's a framework/set of standards that bundles together multiple existing protocols and APIs to enable real-time peer-to-peer communication.

||WebSocket|WebRTCIs|
it one protocol?|✅ Yes — a single protocol (RFC 6455)|❌ No — a framework combining many protocols|
Defined by|IETF (single RFC)|W3C (APIs) + IETF (multiple protocols)|
Underlying transport|TCP|Typically UDP (via SRTP/DTLS), with ICE/STUN/TURN for connectivity|

### Why WebSocket Was Created
 
Before WebSocket, real-time-ish behavior was faked using:
- **Short polling** — the client repeatedly asking the server for updates.
- **Long polling** —  the client sends a request and the server purposefully holds the connection open, server responding only when new data becomes available, after which the client must immediately send a brand new request to repeat the process(reuest- response cycle).

**Problems with both:** too many requests, high server load, and delays.
 
WebSocket solves this by keeping a single connection open and letting the server send data instantly, without the client asking again and again.

### When to Use It
 
Use the WebSocket API when the application needs **instant, two-way communication** between browser and server. Typical use cases:
 
- Chat systems
- Video or live interaction apps
- Multiplayer games
- Collaborative editing (shared documents)
- Live dashboards and visualizations
- Event update systems


### When NOT to Use WebSocket
 
Avoid WebSocket when:
- Data is static
- Requests happen rarely
Keeping a connection open in these cases just wastes server resources.

### Video calling in browser - WebRTC and WebSocket
The two pieces
1. WebRTC — carries the actual audio/video

    * Once two browsers are connected, the actual video and audio stream flows directly between them (peer-to-peer), not through your server.
    * Runs mostly over UDP, optimized for low latency — a bit of dropped video data is fine; nobody wants delay.
    * Handles the media capture (camera/mic), encoding, and the direct peer connection.

2. WebSocket — handles signaling (the "let's connect" handshake)

* Before two browsers can talk directly via WebRTC, they need to exchange some setup information: "here's my video format," "here's my network address," "I want to call you."
* This exchange is called signaling, and WebRTC does not define how to do it — it's left up to the developer.
* WebSocket is a common (but not the only) way to implement signaling, because it's a persistent connection that lets your server relay these setup messages between the two browsers in real time.
 
 ## 2. HTTP vs WebSocket
 
### What Is HTTP?
 
HTTP is a **request → response protocol**.
 
```
Client → Request → Server
Client ← Response ← Server
❌ Connection usually ends
```
 
**Key ideas:**
- Client always starts the exchange.
- Server only replies.
- No continuous communication.
- Server **cannot push data** to the client without a prior request (without tricks like polling).
This is why HTTP is called **unidirectional** (one-way per request).

### What Is WebSocket?
 
WebSocket is a **persistent, two-way** communication channel.
 
```
Client ↔ Server (both can send anytime)
✅ Connection stays open
```
 
**Key ideas:**
- After the connection opens, both sides can talk freely.
- No need to send a new request every time.
This is why WebSocket is called **full-duplex**.


### Core Differences
 
| Aspect | HTTP | WebSocket |
|---|---|---|
| **Communication style** | Client → Server only; server can't push without a request | Client ↔ Server, both directions, anytime |
| **Connection behavior** | Opens → responds → closes | Opens once → stays open |
| **Overhead** | Sends full headers every request (heavy) | Small frames after handshake (lightweight) |
| **Latency** | Slower — new request/response cycle each time | Faster — no repeated setup |
| **Scalability** | Easier (stateless) — supports vertical & horizontal scaling well | Harder (stateful) — needs sticky sessions or shared state (e.g., Redis) for horizontal scaling |
| **Typical use cases** | Static pages, normal APIs, file downloads, one-time data | Chat apps, live notifications, online games, stock updates |

### Relationship Between HTTP and WebSocket
 
WebSocket connections **start as HTTP** and then **upgrade**:
 
```
HTTP request → Upgrade → WebSocket connection
```
 
So they are different protocols, but WebSocket **uses HTTP to start** the connection. They are not competitors — WebSocket is **not a replacement** for HTTP/REST; modern apps often use both together (REST for normal data, WebSocket for real-time features).
### One-Line Summary
 
**HTTP = ask and get once.**
**WebSocket = stay connected and keep talking.**

## 3. Does WebSocket Use TCP?
 
**Yes — WebSocket runs on top of TCP, not UDP.**
 
### How It Works
 
**1. Starts with HTTP (over TCP):**
 
Client sends:
```
Upgrade: websocket
```
 
Server responds:
```
HTTP/1.1 101 Switching Protocols
```
 
This handshake happens entirely over **TCP**.
 
**2. After the upgrade:**
- The connection becomes a **WebSocket** connection.
- It still uses the **same underlying TCP connection** — no new connection is opened.


```
WebSocket = HTTP upgrade + persistent TCP connection
```

### Why TCP?
 
WebSocket needs:
- ✅ Reliable delivery
- ✅ Ordered messages
- ✅ No data loss
TCP provides exactly this.

### Ports
 
| Scheme | Port |
|---|---|
| `http://` | 80 |
| `ws://` | 80 |
| `https://` | 443 |
| `wss://` (secure WebSocket) | 443 |
 
WebSocket runs **on top of existing HTTP infrastructure**, which is why it reuses HTTP's default ports.

### WebSocket vs WebRTC (Transport-Level Comparison)
 
| | WebSocket | WebRTC |
|---|---|---|
| **Protocol** | TCP | UDP (mostly) |
| **Speed** | Moderate | Very fast |
| **Reliability** | High | Lower (but real-time-friendly) |
| **Use case** | Chat, notifications | Video/audio calls |
 
**Analogy:**
- WebSocket (TCP) = sending messages with confirmation 📩
- WebRTC (UDP) = a live conversation 🎤

## 4. Is WebSocket Only Used on the Web?
 
**No — WebSocket is a protocol, not just a browser feature.**
 
It's a common misunderstanding that WebSockets are web-only, because:
- They started in browsers (JavaScript).
- They're heavily used in web apps (chat, live updates).
**Reality:** any application that can open a TCP connection and speak the WebSocket protocol can use WebSockets.
 
### Where WebSockets Are Used
 
- **Web browsers** — chat apps, notifications, live dashboards.
- **Mobile apps (Android/iOS)** — real-time messaging, ride-tracking apps (e.g., Uber-style live location).
- **Backend services** — microservice communication, real-time data pipelines. Example libraries: Python (`asyncio`, `websockets`), Node.js (`ws`, `socket.io`).
- **Games** — multiplayer games, live player position updates.
- **IoT devices** — smart devices and sensors streaming live data.

### WebSocket's framing overhead compared to HTTP
 
WebSocket reduces overhead  by exchanging mostly just message data — not exactly "only" message data, but very close to it, with minimal extra framing overhead compared to HTTP's full headers on every request fpor message data.
* Http has more overhead for sending message data.

### Can WebSocket carry audio/video?
**Technically yes, but it's not what it's designed or used for.**
WebSocket can send two kinds of data:

* Text frames (usually UTF-8 / JSON)
* Binary frames (raw bytes — could be an image, a chunk of audio, anything)

So in theory, you could chop up audio/video into binary chunks and send them frame-by-frame over WebSocket. Some applications do use WebSocket for low-volume audio/video-adjacent tasks (like sending periodic snapshots, or low-bitrate audio clips).

**But for real-time, continuous audio/video calls, WebSocket is a poor fit for audio/video**,becasue
|Issue|Why it hurts video/audio|
|Runs over TCP|TCP guarantees delivery and order — so if one packet is lost, it waits and retransmits before delivering anything newer. For live video, a late frame is useless; you'd rather drop it and move on.|
|No built-in media optimization|No native support for things like jitter buffering, codec negotiation, or adaptive bitrate|
|Goes through a server|All data is relayed via your server — adds latency and bandwidth cost compared to direct peer-to-peer|

That's exactly why WebRTC exists — it uses UDP (via SRTP), which tolerates dropped packets gracefully (a skipped video frame is fine — better than freezing to wait for it), and it connects peer-to-peer, skipping the server entirely for media.

## 5. REST API vs WebSocket API
 
### Core Models
 
- **REST API** — uses the HTTP **request–response model**: client sends a request, server sends a response, connection ends.
- **WebSocket API** — uses a **persistent, full-duplex connection**: one connection stays open, and both client and server can send data anytime.
### Communication Style
 
```
REST:      Client → Request → Server → Response → Done   (one-time communication)
WebSocket: Client ↔ Server (continuous)                    (real-time communication)
```

### Detailed Comparison
 
| Feature | REST API | WebSocket API |
|---|---|---|
| **State** | Stateless (no stored data) | Stateful (stores connection data) |
| **Direction** | Unidirectional (either server or client per request) | Bidirectional |
| **Model** | Request–response | Full-duplex |
| **Headers/Overhead** | HTTP headers sent with every request | No repeated header overhead once connected |
| **Connection** | New TCP connection per request | Single persistent connection |
| **What identifies a "request"** | HTTP Method + path, fresh each time | The connection itself, identified by client IP/port, stays the same for the whole session |
| **Speed** | Slower | Faster |
| **Buffering** | No memory/buffer needed to hold data | Needs memory/buffers to manage live data |
| **Scaling** | Supports vertical and horizontal scaling easily | Mainly easy for vertical scaling; horizontal scaling is harder(bcz of stateful) |

### Advantages and Disadvantages
 
**REST API**
- ✅ Advantages: Simplicity, flexibility, caching, statelessness.
- ❌ Disadvantages: Limited real-time capability; performance overhead since each request/response must carry all necessary info; can get complex for large apps.
**WebSocket API**
- ✅ Advantages: Real-time communication, efficient (one persistent bidirectional connection), scalable (one server can support thousands of websocket connections, bcz of websocket is lightweight), low overhead (single connection for ongoing transmission).
- ❌ Disadvantages: More complex to design and implement than REST; not supported in all environments; vulnerable to security issues if not properly secured.
**Similarities:** Both are used for web applications between client and server, can be customized, and can be secured using authentication and encryption.


### Use-Case Summary
 
| Best for REST | Best for WebSocket |
|---|---|
| Fetching data occasionally | Real-time apps |
| CRUD operations (GET, POST, PUT, DELETE) | Continuous updates |
| User data, product lists | Chat apps, live stock prices, online games |
 
### Simple Analogy
 
- **REST** → like sending an email 📧: send → wait → receive → done.
- **WebSocket** → like a phone call 📞: stay connected, talk anytime.

- **REST** → like sending an email 📧: send → wait → receive → done.

### Important Insight
 
WebSocket is **not a replacement** for REST:
- REST = simple, scalable.
- WebSocket = fast, real-time.
Modern applications often use **both together** — REST for standard data operations and WebSocket for real-time features.
 
**Final answer:** REST API uses a request–response model over HTTP, where each interaction is independent, while WebSocket API maintains a persistent connection allowing real-time, two-way communication between client and server.

## 6. Scaling Concepts (Vertical vs Horizontal)
 
### Vertical Scaling
 
Increasing the power of **one server** (more CPU, more RAM):
```
1 server → stronger server
```
 
### Horizontal Scaling
 
Adding **more servers**, distributing load through a **load balancer**.
 
```
        Users
          ↓
    Load Balancer
     ↙    ↓    ↘
 Server1 Server2 Server3
```
 
**Example:** 1,000 users on 1 server would overload it. Spread across 5 servers, each handles ~200 users — faster and more stable.
 
**Common load balancers:** Nginx, HAProxy, AWS ELB/ALB.
 
### Scaling by Protocol
 
| | HTTP | WebSocket |
|---|---|---|
| Vertical scaling | ✅ Supported | ✅ Supported |
| Horizontal scaling | ✅ Easy | ⚠️ Harder |
| Why | Stateless — each request is independent, so a load balancer can send any request to any server | Stateful — the connection stays open and the server "remembers" the client, requiring **sticky sessions** or a **shared system** (e.g., Redis, message broker) to coordinate across servers |

## 7. WebSocket vs WebRTC
 
### Core Idea
 
- **WebSocket** — **client ↔ server** communication. Goes through a server, persistent connection, used for general real-time apps.
- **WebRTC** — **peer-to-peer (P2P)** communication. Direct **client ↔ client**; the server is only used for initial setup (signaling). Designed for media (audio/video).
### Deep Comparison
 
| Aspect | WebSocket | WebRTC |
|---|---|---|
| **Architecture** | Client ↔ Server ↔ Client (all data goes through the server) | Client ↔ Client (direct; server not used after setup) |
| **Protocol** | TCP | UDP (mostly) |
| **Latency** | Low | Ultra-low (no middle server, uses UDP) |
| **Data type** | Any — text, JSON, binary | Optimized for audio, video, live streaming |
| **Reliability** | Reliable (TCP guarantees delivery) | May drop packets (UDP) — acceptable for live video/audio |
| **Scalability(server load grows as more users join)** | Server handles all connections; more users = more server load (one persistent connection per client) | Less server load (P2P), but hard to scale to large groups since many direct client-to-client connections are needed |
| **Setup complexity** | Simple | Complex — needs ICE candidates, STUN/TURN servers |

### Common Misconception: "WebRTC Is Connectionless Because It Uses UDP"
 
**Not correct.**
 
- **UDP is connectionless.**
- **WebRTC is *not* connectionless** — even though it uses UDP, it builds a connection-like system on top of it.
WebRTC adds, on top of raw UDP:
- Connection setup (**ICE**)
- Session management
- Encryption (**DTLS**)
- Optional reliability for data channels
WebRTC still needs servers for **connection setup (signaling)** and **NAT traversal (STUN/TURN)**. After that, communication is direct.
 
### When to Use What
 
| Use WebSocket for | Use WebRTC for |
|---|---|
| Chat apps 💬 | Video calls 📹 |
| Notifications 🔔 | Voice calls 🎧 |
| Real-time dashboards 📊 | Screen sharing 🖥️ |
 
### Quick Summary Table
 
| | WebSocket | WebRTC |
|---|---|---|
| Communication | Client–Server | Peer-to-Peer |
| Protocol | TCP | UDP |
| Latency | Low | Very low |
| Best for | Data | Audio/Video |
| Complexity | Easy | Complex |
 
**Final answer:** WebSocket is used for real-time communication between client and server over TCP, while WebRTC enables direct peer-to-peer communication (usually over UDP), making it ideal for ultra-low-latency applications like video and voice calls.

## 8. WebSocket Connection Lifecycle (Handshake)
 
### Step-by-Step Flow
 
1. **Client sends an HTTP request** requesting an upgrade:
```
   GET /ws HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade
```
   This is **not** for sending data — it's asking, "Can we switch to WebSocket?"
 
2. **Server responds:**
```
   HTTP/1.1 101 Switching Protocols
```
   This means "Yes, switching to WebSocket."
 
3. **The connection becomes persistent.** HTTP is no longer used for messages — both sides can now send data anytime, with no more request–response cycles.
### Why `ws://` Instead of `http://`?
 
Even though the handshake itself is an HTTP request, using `ws://` tells the browser: *"Start a WebSocket connection (not a normal HTTP request)."* This matters because the browser needs to:
- Add the special `Upgrade` and `Connection` headers.
- Keep the connection open.
- Switch into the WebSocket protocol after the handshake completes.
This only happens automatically when using the `WebSocket()` constructor with a `ws://` or `wss://` URL.
 
### Stateless vs Stateful
 
- **HTTP** is **stateless** — each request is handled independently.
- **WebSocket** is **stateful** — the connection between client and server stays alive until either side terminates it.

## 9. Main WebSocket Events (Browser API)
 
When you create a connection:
 
```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws");
```
 
You can listen for these core lifecycle events in Browser API:
 
| Event | Fires when... | Notes |
|---|---|---|
| **`open`** | Connection is successfully established (after handshake) | Start of communication |
| **`message`** ⭐ | Server sends data to the client | Most-used event — this is how you receive updates |
| **`close`** | Connection is closed | Provides `event.code` (why it closed) and `event.reason` (optional message) |
| **`error`** | Something goes wrong (server unreachable, network issue, protocol error) | Useful for debugging |
 
### Two Equivalent Syntaxes
 
```javascript
// Using addEventListener
socket.addEventListener("open", () => { console.log("Connected"); });
socket.addEventListener("message", (event) => { console.log(event.data); });
 
// Using direct properties (same behavior)
socket.onopen = () => { console.log("Connected"); };
socket.onmessage = (event) => { console.log(event.data); };
```
 
**Summary:**
- `open` → connection ready, start sending.
- `message` → main application logic (receiving data).
- `close` → cleanup or reconnect logic.
- `error` → debugging.
> **Rule:** You can only call `socket.send()` *after* the `open` event has fired.

## 10. WebSocket in FastAPI
 
### Setting Up an Endpoint
 
FastAPI provides a dedicated decorator for WebSocket routes:
 
```python
@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    while True:
        data = await websocket.receive_text()
        await websocket.send_text(data)
```
 
- `@app.websocket("/ws")` defines a **WebSocket endpoint**, not a normal HTTP API route.
- `await websocket.accept()` is the most important line — it accepts the connection. Without it, the connection fails.
- The `while True` loop keeps the connection alive: wait for a message → receive it → send a response.
### Matching Frontend Code
 
```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws");
 
socket.addEventListener("open", () => {
  console.log("Connected");
});
 
socket.send("Hello");
 
socket.onmessage = (event) => {
  console.log(event.data);
};
```
 
### Visual Flow
 
```
Browser                                FastAPI Server
   |                                          |
   |---- connect (/ws) -------------------->  |  new WebSocket('ws://127.0.0.1:8000/ws')
   |<--- accept() --------------------------  |  websocket.accept() — connection established
   |                                          |
   |---- "Hello" --------------------------->  |
   |<--- "Hello" ----------------------------  |
   |                                          |
   |---- "Hi" -------------------------------> |
   |<--- "Hi" --------------------------------  |
```
 
The same connection is reused for every message — no new HTTP requests are made.
 
### Why `async`/`await` Matters
 
WebSocket communication needs continuous waiting and sending. FastAPI uses `async`/`await` so the server is **non-blocking** — it can handle multiple users concurrently without separate threads per connection.
 
### Real-World Use Cases
 
Chat apps, live notifications, stock price updates, multiplayer games — anywhere the server needs to push data without waiting for a request.

## 11. FastAPI Chat App Example (Multi-User Broadcast)
 
### What This Project Does
 
A real-time chat app where:
- Multiple users connect.
- Each user sends messages.
- The server distributes (broadcasts) messages to other connected users.
### High-Level Architecture
 
```
Browser (User A) ─┐
                   ├──> FastAPI Server ───> All Users
Browser (User B) ─┘
```
 
The server acts as a **middleman** that receives and distributes messages.
 
### Frontend Flow
 
```javascript
// 1. Create connection with a unique user ID
const socket = new WebSocket('ws://127.0.0.1:8000/ws/' + id);
 
// 2. On connect
socket.addEventListener('open', () => {
  showMessage('Connected to server.');
});
 
// 3. Sending a message
socket.send(inputValue);
 
// 4. Receiving a message
socket.onmessage = (event) => {
  showMessage(event.data);
};
```
 
### Backend: `ConnectionManager` Class
 
This class tracks all connected clients:
 
```python
class ConnectionManager:
    def __init__(self):
        self.active_connections: List[WebSocket] = []
 
    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)
 
    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)
 
    async def send_personal_message(self, message: str, websocket: WebSocket):
        await websocket.send_text(message)
 
    async def broadcast(self, message: str, websocket: WebSocket):
        for connection in self.active_connections:
            if connection == websocket:
                continue
            await connection.send_text(message)
```
 
| Method | Purpose |
|---|---|
| `connect()` | Accepts the connection and adds the user to the active list |
| `disconnect()` | Removes a user when they leave |
| `send_personal_message()` | Sends a message to one specific user |
| `broadcast()` | Sends a message to all *other* connected users (skips the sender, since the sender already got their own copy) |
 
### WebSocket Endpoint
 
```python
connectionmanager = ConnectionManager()
@app.websocket("/ws/{client_id}")
async def websocket_endpoint(websocket: WebSocket, client_id: int):
    await connectionmanager.connect(websocket)
    try:
        while True:
            data = await websocket.receive_text()
            # Send back to the same user
            await connectionmanager.send_personal_message(f"You : {data}", websocket)
            # Broadcast to everyone else
            await connectionmanager.broadcast(f"Client #{client_id}: {data}", websocket)
    except WebSocketDisconnect:
        connectionmanager.disconnect(websocket)
        await connectionmanager.broadcast(f"Client #{client_id} left", websocket)
```
 
- `WebSocketDisconnect` is the exception raised when a client disconnects (tab closed, network lost, etc.). On disconnect, the user is removed from the connection list and others are notified.
### Full Flow
 

| Step | What happens | Method/event |
|---|---|---|
| 1. User opens the page `ws://127.0.0.1:8000/ws/` | Browser creates the WebSocket and starts the handshake , and the hanshake request (HTTP upgarde) is send to server | `new WebSocket(...)` triggers it |
| 2. Server accepts it | Backend explicitly accepts the connection in hadshake | `await websocket.accept()` |
| 3. Handshake completes | Browser confirms the connection is now open | `socket.onopen` fires (frontend) |
| 4. User sends a message | Frontend sends data over the open socket | `socket.send(...)` |
| 5. Server receives it | Backend's loop picks up the incoming text | `data = await websocket.receive_text()` |
| 6. Server sends it back to sender + broadcasts to others | Two separate sends from the backend | `send_personal_message()` → `websocket.send_text(...)`; `broadcast()` → `connection.send_text(...)` for each other client |
| 7. Repeat indefinitely | The `while True` loop keeps listening for more messages | — |
### Key Concepts
 
- **Persistent connection** — one connection handles many messages; no repeated HTTP requests.
- **Connection list** (`self.active_connections`) — how the server tracks who's currently connected.
- **Broadcast logic** — the core mechanism behind chat apps.
- **Infinite loop** (`while True`) — keeps the connection alive.
### Limitation of example and Real-World Improvement
 
This example stores `active_connections` **in memory**, which means:
- ❌ Not scalable — it only works on a single server instance.
Real-world apps typically use **Redis**, message queues, or pub/sub systems to share connection state across multiple server instances.
 
**One-line summary:** FastAPI chat app = WebSocket connections + a connection list + broadcast messages.

* Where is the `Websocket()`?
  * A browser-based frontend app (React, Vue, plain JS in an HTML page) runs inside the browser, and when it(browser) calls `new WebSocket(...)`, it's using the browser's built-in WebSocket API.
# Who Calls `new WebSocket(...)`?

| Who/what | Role |
|---|---|
| **Browser** | Provides the `WebSocket` class as part of its built-in JavaScript environment (like it also provides `fetch`, `document`, `console`, etc.) |
| **Your code** | Writes `new WebSocket(...)` — this is *your* instruction, executed by the browser's JS engine |
| **Browser (again, after your call)** | Once your code creates the object, the browser's internals take over: it sends the actual handshake request over the network, manages the TCP connection, parses incoming frames, and fires events back to your code (`onopen`, `onmessage`, etc.) |

## 12. WebSocket with Django (Django Channels)
 
### The Problem
 
Plain Django (HTTP) follows: client → request → server → response → connection closed. The server **cannot send data anytime** — it can only reply when asked.
 
With WebSockets: client ⇄ server, connection stays open, enabling real-time communication where the server can push data anytime. Useful for chat apps, live notifications, dashboards.
 
### Why Plain Django Can't Do WebSockets
 
Django, by default, uses **WSGI** (synchronous) — which only supports HTTP. To add WebSocket support, Django needs **Django Channels**, which switches to **ASGI** (asynchronous).
 
### Architecture
 
```
Browser (WebSocket)
       ↓
ASGI Server (Daphne) - started via `daphne project.asgi:application`,
                        which loads the `application` object from asgi.py
       ↓
Django Channels
       ↓
Consumer (the WebSocket equivalent of a view)
```
 
### Step-by-Step Setup
 
**1. Install packages**
```bash
pip install channels daphne
```
- `channels` → adds WebSocket support.
- `daphne` → an ASGI server (like Gunicorn, but async-capable).
**2. `settings.py`**
```python
INSTALLED_APPS = [
    ...,
    'channels',
]
 
ASGI_APPLICATION = 'project.asgi.application'
```
This tells Django to use ASGI instead of WSGI.
 
**3. ASGI configuration (`asgi.py`)** — the core routing decision-maker:
```python
application = ProtocolTypeRouter({
    "http": get_asgi_application(),
    "websocket": URLRouter(
        chat.routing.websocket_urlpatterns
    ),
})
```
 
| Request type | Routed to |
|---|---|
| HTTP | Django views |
| WebSocket | Consumer (the WebSocket "view") |
 
**4. Consumer — the most important part**
 
A **Consumer** is the WebSocket equivalent of a Django view.
 
```python
from channels.generic.websocket import WebsocketConsumer
class ChatConsumer(WebsocketConsumer):
    def connect(self):
        self.accept()
 
    def receive(self, text_data):
        data = json.loads(text_data)
        message = data['message']
        self.send(text_data=json.dumps({'message': message}))
```
 
| Django HTTP | Django Channels Consumer |
|---|---|
| `request` | `receive()` |
| `response` | `send()` |

**Django Channels: `receive()` and `send()`**

| Concept | What it is |
|---|---|
| `receive()` | The method that's called with incoming data from the client — this is how data arrives, same role as `request` |
| `send()` | The method you call to send data back to the client — this is how data leaves, same role as `response` |

**5. Routing (`routing.py`)**
```python
websocket_urlpatterns = [
    path('ws/chat/', ChatConsumer.as_asgi()),
]
```
**6. Frontend (JavaScript)**
```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws/chat/");
 
socket.onmessage = function(e) {
  const data = JSON.parse(e.data);
};
 
socket.send(JSON.stringify({ 'message': message }));
```
 
**7. Run the server**
```bash
daphne project.asgi:application
```
Important: Django's default `runserver` (WSGI) is **not enough** — an ASGI server like Daphne is required.

```
daphne project.asgi:application
         └──┬──┘ └─────┬─────┘
        module path   the variable name
        (project/asgi.py)   (application = ProtocolTypeRouter({...}))
```
 
### Full Request Flow
 
1. Browser opens `ws://127.0.0.1:8000/ws/chat/`. here new WebSocket() will trigger and send hanshake request to server.
2. Request goes to: ASGI → `ProtocolTypeRouter` → `"websocket"` key in ProtocolTypeRouter.
3. Routed to `ChatConsumer`.
4. Lifecycle events map to consumer methods: first accept event happens to the HTTT handshake request.Then other events .
| Event | Consumer method |
|---|---|
| Connect | `connect()` |
| receives data from the client | `receive()` |
| Send response to client | `send()` |

### Key Concepts to Remember
 
- **WebSocket = persistent connection** — no request–response cycle.
- **Consumer = WebSocket handler**, analogous to a Django view.
- **Channels = the bridge** that gives Django WebSocket support.
- **ASGI vs WSGI:**
| | WSGI | ASGI |
|---|---|---|
| Sync | ✅ | ❌ |
| Async | ❌ | ✅ |
| WebSockets | ❌ | ✅ |

## 13. What Is Daphne?
 
**Daphne is an ASGI server** — it runs your Django app and knows how to handle WebSockets, HTTP, and async connections. It's the program that sits between the client and your Django (ASGI) application.
 
### Why Daphne Is Needed
 
**Normal Django (WSGI):**
```
Browser → Gunicorn → Django (WSGI) → Response
```
❌ Only HTTP, no WebSockets.
 
**With Channels + Daphne (ASGI):**
```
Browser → Daphne → Django (ASGI) → Consumer/View
```
✅ Handles HTTP, WebSockets, and async tasks.
 
### What Daphne Does
 
1. **Accepts the connection** — either HTTP or WebSocket.
2. **Understands the protocol type:**
   | Protocol | Routed to |
   |---|---|
   | HTTP | Django views |
   | WebSocket | Consumer |
3. **Keeps the connection open for WebSockets** — the key difference from a traditional WSGI server:
   | Gunicorn | Daphne |
   |---|---|
   | Request → Response → Close | Connect → Stay open → Send/Receive anytime (onlyu for websocket)|

### `ProtocolTypeRouter` — The Decision-Maker
 
Inside `asgi.py`:
```python
application = ProtocolTypeRouter({
    "http": get_asgi_application(),
    "websocket": URLRouter(
        websocket_urlpatterns
    ),
})
```
 
It distinguishes HTTP vs WebSocket by inspecting the incoming request:
 
**HTTP request:**
```
GET /home HTTP/1.1
```
→ Routed to `"http"`.
 
**WebSocket request:**
```
GET /ws/chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
```
The special `Upgrade: websocket` header signals → routed to `"websocket"`.
 
### Full Routing Paths
 
**Case 1 — HTTP request:**
```
Browser → Daphne → asgi.py → ProtocolTypeRouter → "http"
→ get_asgi_application() → Django URL routing → views.py
```
 
**Case 2 — WebSocket request:**
```
Browser → Daphne → asgi.py → ProtocolTypeRouter → "websocket"
→ URLRouter → websocket_urlpatterns → Consumer
```
 
| | HTTP | WebSocket |
|---|---|---|
| Routing file | `urls.py` | `routing.py` |

## 14. FastAPI vs Django Channels: Receiving Messages
 
### Django Channels (Event-Driven)
 
```python
def receive(self, text_data):
    data = json.loads(text_data)
    message = data['message']
```
- `receive()` is **automatically called** by the framework whenever a message arrives.
- `text_data` is passed in for you; you manually parse the JSON.
- Mental model: *"When a message comes in → run this function."*
### FastAPI (Manual Async Loop)
 
```python
data = await websocket.receive_text()
```
- You **explicitly call** `receive_text()`.
- It asynchronously waits for a message and returns the text directly.
- Mental model: *"Keep listening and fetch messages yourself."*
### Core Difference
 
| | Django Channels | FastAPI |
|---|---|---|
| **Style** | Event-based | Async loop |
| **Receiving data** | Framework auto-calls `receive()` | You call `receive_text()` |
| **Control** | Framework-driven | Developer-driven |
 
### Why the Difference Exists
 
- **Django Channels** is built on an event system that handles the lifecycle automatically: `connect()`, `receive()`, `disconnect()`.
- **FastAPI** is built on explicit `async`/`await` control — you manage the loop, receiving, and sending yourself.
### Text vs Binary Data
 
- **Django:** `receive(self, text_data)` only handles text frames by default. Binary data arrives via a separate `bytes_data` parameter.
- **FastAPI:** you explicitly choose what to receive — `await websocket.receive_text()` or `await websocket.receive_bytes()`.

## 15. Sync vs Async in Django Channels -
:::info
This topic is important.
::::
 
### Why Django Examples Often Skip `async`/`await`
 
Because they use a **synchronous consumer**:
```python
class ChatConsumer(WebsocketConsumer):
    def receive(self, text_data):
        self.send(text_data=text_data)
```
No `async`, no `await` — it runs in a **thread (blocking style)**.
 
### Django Channels Supports Two Consumer Types
 
**1. Sync Consumer**
```python
from channels.generic.websocket import WebsocketConsumer
 
class ChatConsumer(WebsocketConsumer):
    def receive(self, text_data):
        self.send(text_data=text_data)
```
 
**2. Async Consumer**
```python
from channels.generic.websocket import AsyncWebsocketConsumer
 
class ChatConsumer(AsyncWebsocketConsumer):
    async def receive(self, text_data):
        await self.send(text_data=text_data)
```
 
### Why Tutorials Often Use the Sync Version
 
- **Simplicity** — easier to understand, no need to think about async yet.
- **Compatibility** — Django's ORM is mostly synchronous (`User.objects.all()`), which works easily in a sync consumer.
### Why FastAPI Always Uses Async
 
FastAPI is built on ASGI with an **async-first design** — there is no synchronous option, unlike Django.
 
```python
async def endpoint():
    await something()
```
 
### Comparison
 
| | Django Channels | FastAPI |
|---|---|---|
| Sync support | ✅ | ❌ |
| Async support | ✅ | ✅ |
| Default style | Sync | Async |
 
### Important Rule for Async Consumers
 
If using `AsyncWebsocketConsumer`, **never** run blocking ORM calls directly:
 
```python
# ❌ BAD — blocks the event loop
User.objects.all()
```
 
Instead, wrap blocking calls:
 
```python
# ✅ GOOD
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
### When to Use Async Consumers
 
Use `AsyncWebsocketConsumer` when you have:
- Many concurrent users (e.g., chat apps)
- Real-time updates
- External API calls
- Non-blocking task requirements
**One-line clarity:** FastAPI = "everything async." Django = "you choose sync or async."

## 16. Async Database Access in Django and FastAPI

:::info
This topic is imp
::::
 
### Why DB Queries Aren't Naturally `await`-able in Django
 
Django's ORM is **synchronous (blocking)**. A call like:
```python
User.objects.all()
```
blocks the thread until the database responds. You **cannot** simply write:
```python
await User.objects.all()  # ❌ ERROR — not awaitable
```
because it's not an async function and doesn't return a coroutine.
 
**What actually happens during a sync query:**
```
Python → DB driver → Database → wait → response → return
```
During this wait, the thread is **blocked** and can't handle other requests.
 
### Using the ORM From Async Code
 
```python
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
Internally, this moves the synchronous call to a thread pool, runs the blocking DB query there, and returns the result without blocking the main async event loop. The async app itself doesn't block — but the DB call is still sync *internally*.

```
Your async event loop (main thread)
        ↓
sync_to_async grabs a separate thread from a thread pool
        ↓
Runs User.objects.all() on THAT thread  ← still fully synchronous/blocking, right here
        ↓
That thread blocks while waiting for the DB
        ↓
Once it's done, the result is handed back to your async event loop
        ↓
Your main event loop was free to do other things the whole time

```
* Term Meaning-"Doesn't block" (the async app)-Your main event loop stays free to handle other tasks while the DB call happens elsewhere.
* Term Meaning"Sync internally" (the DB call)- The actual database query is still a normal blocking call — it just happens on a separate worker thread instead of the main thread

* The blocking is real — it(`sync_to_async`) didn't get magically removed. It just got moved to a thread that isn't doing anything else important, so its blockage doesn't stall the rest of your app.

### Why Django's ORM Isn't Async
 
1. **Database drivers** — most traditional drivers (e.g., `psycopg2` for PostgreSQL) are blocking/sync by design. (A database driver is a library that lets your Python code talk to a database.)
2. **Framework design** — Django was built before async became popular, and its entire ORM is designed around synchronous execution.
### Async Database Access in FastAPI
 
FastAPI can do fully async DB access using specialized async-native libraries:
 

| Library | What it actually is | Talks directly to the database? |
|---|---|---|
| `asyncpg` | A true DB driver — written specifically for async PostgreSQL access | ✅ Yes, directly |
| `asyncmy` | A true DB driver — async MySQL driver | ✅ Yes, directly |
| `databases` | A query library / thin wrapper — not a driver itself | ❌ No — it sits on top of drivers like `asyncpg` |
| SQLAlchemy (async mode) | A full ORM — also not a driver itself | ❌ No — it also relies on an underlying driver to actually talk to the DB |

These are:
- ✅ Fully asynchronous
- ✅ Work with `async`/`await`
- ✅ Non-blocking
- ✅ Designed for high concurrency
**SQLAlchemy** is a **hybrid ORM** — originally synchronous, with async support added in version 1.4+.
 
```python
result = await database.fetch_all(query)
```
This is truly non-blocking, using an async DB driver.
 
### Side-by-Side
 
| | Django ORM | Async-native (e.g., SQLAlchemy async, asyncpg) |
|---|---|---|
| Async support | ❌ | ✅ |
| Blocking | ✅ | ❌ |
| Needs wrapper (`sync_to_async`) | ✅ | No |
 
### Common Mistake
 
```python
class ChatConsumer(AsyncWebsocketConsumer):
    async def receive(self, text_data):
        users = User.objects.all()  # ❌ BAD — blocks the event loop, kills performance
```
 
**Correct approach:**
```python
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
### When Sync Django Is Actually Fine
 
If the app is mostly **DB-heavy with standard CRUD operations(std crud operations mostly have noraml traffic)**, a sync Django setup is often **simpler and perfectly adequate** — async only pays off under high concurrency or many simultaneous long-lived connections.

* Async doesn't make a single DB-heavy query run faster — the database still takes the same time to do its work. Async's benefit is letting your app stay responsive to other requests while waiting on that DB time. So "DB-heavy but normal traffic" doesn't automatically benefit from async — only "DB-heavy and lots of simultaneous waiting requests" does.ASync helps if there simultaneous waiting requests case , not with normal traffic for "DB heavy " operations.

## 17. Authentication Fundamentals
 
**Authentication** answers: *"How does the server know who you are?"* (e.g., after you log in, the server must remember you for future requests.)
 
There are two major approaches:
 
### A. Session-Based Authentication (Traditional Django)
 
**Idea:** The server stores your login info; the browser only holds a reference to it.
 
**Flow:**
1. **Login:** username + password → server.
2. **Server creates a session:** `session_id = "abc123"`, stored server-side (DB/cache).
3. **Server sends a cookie:** `Set-Cookie: sessionid=abc123`.
4. **Browser sends the cookie on every request:** `Cookie: sessionid=abc123`.
5. **Server looks up the session:** `session_id → user`.
**Where data lives:**
 
| Location | Stores |
|---|---|
| Browser | `session_id` (cookie only) |
| Server | Full user session data |
 
**Pros:** Very secure (server controls state); easy in Django via `request.user`; built-in support.
**Cons:** Needs server memory for sessions; harder to scale across multiple servers; not ideal for APIs/mobile apps.
 
### How Django Implements This by Default
 
In `settings.py`:
```python
MIDDLEWARE = [
    ...,
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
]
```
 
Authentication is **not** done by `SessionMiddleware` alone — it's a combination of two middlewares:
 
| Middleware | Job |
|---|---|
| `SessionMiddleware` | Reads the `sessionid` cookie, loads session data from server, attaches `request.session` |
| `AuthenticationMiddleware` | Reads `request.session`, gets the `user_id`, fetches the user, attaches `request.user` |
 
**Flow:**
```
Browser sends cookie: sessionid=abc123
        ↓
SessionMiddleware → request.session = session_data
        ↓
AuthenticationMiddleware → request.user = User object
```

* Session data is just a key-value store in server,tied to a specific user's browsing session.it is like `_auth_user_id: 5`, `_auth_user_hash: "..."`,`request.session["theme"] = "dark"` etc.

You only need to write custom code when doing custom authentication (JWT, tokens), WebSocket auth, or custom user logic.
 
**Django Channels equivalent:** WebSocket connections in Django Channels don't have a request object — they have a scope instead. So Django's normal HTTP-only middlewares (SessionMiddleware, AuthenticationMiddleware) can't be used directly — they're written specifically to work with request, not scope.
* **Django Channels** provides **AuthMiddlewareStack** — think of it as: "the WebSocket-compatible version of SessionMiddleware + AuthenticationMiddleware, bundled together as one thing."

### Sessions vs Cookies — Core Relationship
 
- A **session** is data stored on the **server** to remember a user between requests (e.g., `request.session["user_id"] = 5`), stored in a database or cache (like Redis).
- A **cookie** is a small piece of data — typically just the `session_id` — stored in the **browser**.
**Key rule:**
- Cookie stores only the `session_id`.
- Session (server-side) stores the actual user data.
Cookie ≠ Session. The cookie is just a **key**; the session is the **data**.
 
**Full flow:**
1. User logs in (username + password → server).
2. Server creates a session: `{"abc123": {"user_id": 5}}`.
3. Server sends `Set-Cookie: sessionid=abc123` — stored in the browser.
4. On the next request, the browser automatically sends `Cookie: sessionid=abc123` (sent as an HTTP **request header**, e.g., `Cookie: sessionid=abc123; theme=dark`).
5. Server looks up `abc123 → user` and sets `request.user`.
In Django, `request.session` is a Python object (interface) used to read/write session data —
example:
```
request.session["name"] = "John"
1. You write this line in your view
        ↓
2. request.session (the interface object) receives this write
        ↓
3. Internally, it marks "name" → "John" as a change to be saved
        ↓
4. At the end of the request, Django's SessionMiddleware
   takes request.session's data and SAVES it into the
   real storage (DB row / Redis entry) tied to that session_id

```
### B. Token-Based Authentication (Modern APIs)
 
**Idea:** The server does **not** store session state — the client holds everything (the token).
 
**Flow:**
1. **Login:** username + password → server.
2. **Server generates a token:** `token = "xyz123abc"`.
3. **Server sends the token:** `{ "token": "xyz123abc" }`.
4. **Client stores the token** (local storage / memory / app storage).
5. **Client sends the token on every request:** `Authorization: Token xyz123abc`.
6. **Server verifies the token:** `token → user`.
**Where data lives:**
 
| Location | Stores |
|---|---|
| Client | The token |
| Server | Usually nothing, or a minimal token→user mapping |
 
**Pros:** Stateless (no server memory needed); scales easily; best for APIs and mobile apps.
**Cons:** Slightly less secure if mishandled; tokens can be stolen; requires expiry handling.
 
### Two Ways to Implement Token Auth
 
**1. Token Stored in a Database (e.g., Django REST Framework's Simple Token Auth)**
 
The server *does* store a token → user mapping:
1. User logs in → server creates token `abc123`.
2. Server stores `abc123 → user_id = 5`.
3. Client sends `Authorization: Token abc123`.
4. Server looks up `abc123` in the DB → finds `user_id = 5`.
```python
Token.objects.get(key=token)  # DB lookup
```
 
| Location | Stores |
|---|---|
| Client | Token |
| Server | Token → user mapping |
 
**2. JWT (JSON Web Token) — No DB Lookup Needed**
 
The token itself **contains** the user data (encoded, not stored server-side):
1. Server creates: `token = jwt.encode(payload, SECRET_KEY, algorithm="HS256")`. #payload={"user_id":--,"exp":--}
2. Client sends: `Authorization: Bearer <JWT>`.
3. Server **decodes** the token directly: `user_id = 5`. No database lookup required.
```python
jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
```
 
| Location | Stores |
|---|---|
| Client | Token |
| Server | Only the secret key (used for verification) |
 
### DB Token vs JWT
 
| | DB-backed token | JWT |
|---|---|---|
| Server storage | ✅ Yes | ❌ No |
| Lookup needed | ✅ Yes | ❌ No |
| Speed | Slower | Faster |
| Revocation | Easy (delete from DB) | Hard (token is valid until it expires) |
 
**Analogy:**
- DB token = an ID number; the server checks a register.
- JWT = an ID card; the server just verifies the signature.

### JWT Signing Algorithms
 
**Symmetric (e.g., HS256):**
- Same `SECRET_KEY` used to both **encode** and **decode**.
- The secret key is saved on the server only.
**Asymmetric (e.g., RS256):**
 
| Operation | Key used |
|---|---|
| Encode (sign) | Private key |
| Decode (verify) | Public key |
 
```python
# Encode
jwt.encode(payload, PRIVATE_KEY, algorithm="RS256")
 
# Decode
jwt.decode(token, PUBLIC_KEY, algorithms=["RS256"])
```

| Algorithm | Best suited for |
|---|---|
| HS256 (shared secret) | Simple apps, single-server Django setups |
| RS256 (public/private key pair) | Large systems, microservices (verification can happen anywhere with just the public key) |
 
### Session Auth vs Token Auth — Summary
 
| | Session-Based | Token-Based |
|---|---|---|
| **Storage** | Server | Client |
| **State** | Stateful | Stateless |
| **Scaling** | Harder | Easier |
| **Typical usage** | Django templates / traditional web apps | APIs / mobile apps |
| **Carrier** | `sessionid` cookie | `Authorization` header |

## 18. WebSocket Authentication
 
### The Core Problem
 
Normal HTTP APIs send a token via a header:
```
Authorization: Token abc123
```
But **WebSocket protocol, especially from browsers, cannot send custom headers** during the handshake — so authentication needs a workaround.

* **It's not a limitation of the WebSocket protocol itself** — **it's a limitation of the browser's JavaScript `WebSocket API`**. The websocket protocol can carry custom headers; browsers just don't give you a way to set them.

WebSocket has **no standard built-in auth system** like HTTP does, so different applications use different workarounds during the handshake phase (HTTP → WebSocket upgrade).
 
### Common Authentication Methods
 
| Method | How it works | Pros | Cons |
|---|---|---|---|
| **JWT in `Authorization` header**
(applicable in -- HTTP, websocket -outside of browser)  | `Authorization: Bearer <token>` | Most secure, clean architecture, used in production backend systems; works in Node.js, Python (Django Channels, FastAPI) | **Browsers don't allow setting this header on WebSocket connections (HTTP request it does)** — for websocket connections: only works outside the browser  (mobile apps, backend-to-backend, tools like Postman) |
| **JWT in query parameters**(applicable http,websocket) | `ws://localhost:8000/ws?token=<jwt>` | Very widely used, works in browsers, easy to implement | Token is visible in URL, browser history, and server logs. Mitigated with short-lived or one-time tokens |
| **Cookies / session-based auth**(applicable http,websocket) | Browser automatically sends the session cookie during the WebSocket handshake | Very common in Django; secure if combined with "HTTPS + `HttpOnly` cookies"; zero extra effort from the developer | Mainly convenient for browser-based web apps |
| **First-message authentication** | Connect first, then send `{"token": "abc123"}` as the first message; server validates it | Works in any environment | Connection is open before authentication completes — slight security risk; less preferred |

**For session based auth - The protections**

| Protection | What it guards against |
|---|---|
| **HTTPS** | Without it, the cookie travels over the network in plain text — anyone snooping on the network (e.g., on public WiFi) could read the `sessionid` cookie as it's sent and steal it. HTTPS encrypts the connection, so the cookie can't be read in transit. |
| **`HttpOnly` flag** | Without it, malicious JavaScript injected into your page (via an XSS attack) could read the cookie using `document.cookie` and steal it. `HttpOnly` blocks JavaScript from reading the cookie at all — only the browser (sending it automatically) and the server (reading it) can access it. |

### Headers That *Do* Work Automatically in Browser WebSocket Handshakes
 
Even though custom headers like `Authorization` are blocked, browsers still automatically send:
- `Cookie`
- `Origin`
- `Host`
This is exactly why **session-based (cookie) auth works** in browser WebSocket connections, while a manually-set `Authorization` header does not.

### What Companies Actually Use (Practical Guidance)
 
| Scenario | Recommended approach |
|---|---|
| Django web app | Cookies (session auth) — simplest, built-in |
| SPA (single page app) (e.g., React + API) | JWT via query param or other workaround, since the `Authorization` header isn't available in-browser |
| Mobile app | JWT in `Authorization` headers (headers work fine outside the browser) |
| High-security systems | Short-lived JWTs with a refresh-token flow, possibly combined with cookies |

**Realistic recommendation for "a good web app"**
For a solid, real-world FastAPI + React setup:

1. **Access token (JWT)** — short-lived (e.g., 15 minutes), kept in memory on the React side, sent via Authorization header for normal API calls.
2. **Refresh token** — longer-lived, stored in an HttpOnly, Secure, SameSite cookie — used only to silently get a new access token when the old one expires.
3. **WebSocket connections** — since headers aren't available in websocket in browser, pass the short-lived access token via query parameter at connection time (acceptable specifically because it's short-lived — limiting the exposure window from being logged/visible).

## 19. Cookies vs Tokens: Browsers vs Mobile Apps
 
### Do Cookies Only Work in Browsers?
 
**No.** Cookies work in **any HTTP client** — including mobile apps, backend services, and CLI tools.
 
The real distinction is **automatic handling**:
- **Browsers** handle cookies automatically: they store `Set-Cookie` responses and automatically attach the right `Cookie` header on every subsequent request — including the WebSocket handshake. Zero developer effort.
- **Mobile apps** do **not** handle cookies automatically (unless using special libraries). The cookie must be stored and attached manually.


# `Set-Cookie` vs `Cookie` Headers

| Header | Sent by | Direction | Purpose |
|---|---|---|---|
| **`Set-Cookie`** | Server | Server → Browser (in a response) | Server tells the browser: "store this cookie" |
| **`Cookie`** | Browser | Browser → Server (in a request) | Browser tells the server: "here's the cookie I'm holding" |


### Why Mobile Apps Prefer Tokens
 
Since cookies aren't automatic on mobile, it's simpler to use:
```
Authorization: Bearer <JWT>
```
This works consistently across HTTP, WebSockets, and general API calls, with no dependency on a cookie system in mobile apps.Unlike browsers, mobile apps **can** freely send custom headers — which is why mobile WebSocket clients can use `Authorization: Bearer <token>` directly, while browser-based WebSocket clients cannot.
 
| | Browser | Mobile app |
|---|---|---|
| Cookie handling | Automatic | Manual / requires extra libraries |
| Custom header support (e.g., `Authorization`) on WebSocket handshake | ❌ Not allowed | ✅ Allowed |
| Common auth choice in websocket | Cookies, or JWT via query param | JWT in `Authorization` header |

### Can You Send a JWT Inside a Cookie?
 
**Yes, absolutely.** A cookie is just generic key–value storage — it is **not** limited to holding only a `session_id`. The server can set:
```
Set-Cookie: access_token=eyJhbGciOiJIUzI1NiIs...; HttpOnly; Secure; SameSite=Lax
```
The browser will store it and automatically send it on future requests, including the WebSocket handshake.
 
**Why people assume cookies = session_id only:** In frameworks like Django, the cookie conventionally stores just a `session_id`, while the actual user data lives server-side. That convention is what's called **session-based authentication** — but it's a convention, not a hard limitation of cookies themselves.


### Session-in-Cookie vs JWT-in-Cookie
 
| | Session (Django default) | JWT in cookie |
|---|---|---|
| Cookie content | `sessionid=abc123` | `access_token=<JWT>` |
| Server storage | Stores full user data, keyed by session ID | No storage needed — token is self-contained |
| State | Stateful | Stateless |

### Is Storing a JWT in a Cookie a Problem?
 
Not inherently — many companies do this. But there are trade-offs:
 
- **CSRF risk:** Cookies are sent automatically with every matching request, which means an attacker's site could trigger unwanted requests to your server (Cross-Site Request Forgery).
- **Less control vs. headers:** With an `Authorization` header, the developer explicitly controls when the token is sent. With cookies, it's sent automatically — sometimes when not wanted.

### Who Can Access an `HttpOnly` Cookie?
 
| Actor | Access? |
|---|---|
| JavaScript (frontend, via `document.cookie`) | ❌ No |
| Browser (automatic sending in requests) | ✅ Yes |
| Your backend server (e.g., `request.COOKIES.get("access_token")` in Django) | ✅ Yes |
| Malicious injected JavaScript (XSS attack) | ❌ No |
| Other websites (cross-site requests, when `SameSite=Strict`) | ❌ No |
 
This is precisely the protection `HttpOnly` and `SameSite` are designed to provide.

## 20. WebSocket Authentication in Django Channels (Token/JWT Flow)
 
### The Problem Being Solved
 
In normal APIs, a token is sent via a header. In WebSockets, headers generally **can't** be used for this (browser limitation) — so authentication needs a different mechanism.
 
**Core idea:** since headers don't work, the **token is sent via the URL's query string**, and the server **manually authenticates** the user from it.
 
### Full Beginner-Level Flow
 
**Step 1 — User logs in (plain HTTP):**
```
POST /api/login
```
Server checks the username/password and returns:
```json
{ "token": "abc123" }
```
 
**Step 2 — Create the WebSocket connection with the token in the query string:**
```javascript
new WebSocket("ws://localhost:8000/?token=abc123");
```
 
**Step 3 — WebSocket handshake carries the token:**
```
GET /?token=abc123
Upgrade: websocket
```
The server receives this as `scope["query_string"]`.

### Custom Middleware (Core Authentication Logic)
 
```python
from rest_framework.authtoken.models import Token
from urllib.parse import parse_qs
from channels.db import database_sync_to_async
from django.contrib.auth.models import AnonymousUser
 
@database_sync_to_async
def returnUser(token_string):
    try:
        user = Token.objects.get(key=token_string).user
    except:
        user = AnonymousUser()
    return user
 
class TokenAuthMiddleWare:
    def __init__(self, app):
        self.app = app
 
    async def __call__(self, scope, receive, send):
        query_string = scope["query_string"]
        query_params = query_string.decode()
        query_dict = parse_qs(query_params)
        token = query_dict["token"][0]
        user = await returnUser(token)
        scope["user"] = user
        return await self.app(scope, receive, send)
```
 
This middleware:
1. Extracts the token from `scope["query_string"]`.
2. Looks up the user from the token via `Token.objects.get(key=token).user`.
3. Attaches the result to `scope["user"]`.
This mirrors exactly what Django normally does for session-based authentication — just adapted for WebSockets.
 
`@database_sync_to_async` is required because Django's ORM is synchronous while Channels is async — this decorator safely runs the blocking DB call without blocking the event loop.
 
### Wiring the Middleware into Routing
 
```python
application = ProtocolTypeRouter({
    "websocket": TokenAuthMiddleWare(
        AllowedHostsOriginValidator(
            URLRouter(
                [path("", TokenAuthConsumer.as_asgi())]
            )
        )
    )
})
```
 
| Component | Role |
|---|---|
| `ProtocolTypeRouter` | Decides HTTP vs WebSocket |
| `TokenAuthMiddleWare` | Adds authentication (sets `scope["user"]`) |
| `URLRouter` | Routes WebSocket paths to the correct consumer |
 
Internally, `self.app` in the middleware refers to the wrapped ASGI callable (here, the `URLRouter`), which is why `await self.app(scope, receive, send)` works — it passes control down the chain.
 
### Using the Authenticated User in the Consumer
 
```python
class TokenAuthConsumer(AsyncJsonWebsocketConsumer):
    async def connect(self):
        await self.accept()
        print(self.scope["user"].username)
```
 
By the time `connect()` runs, `self.scope["user"]` is already populated by the middleware — no extra lookup needed.
 
### Why Not Just Authenticate Inside `connect()`?
 
You *can* technically do manual authentication directly inside `connect()`:
```python
async def connect(self):
    token = ...  # read manually
    user = validate_token(token)
```
But this is **not recommended**. Using middleware instead is better because it's:
- Reusable across multiple consumers.
- A clean separation of concerns.
- Consistent — works the same way for every consumer in the app.
### Roles: Middleware vs `connect()`
 
| Stage | Responsibility |
|---|---|
| **Middleware** | **Authentication** — establishing *who* the user is (`scope["user"] = user`) |
| **`connect()`** | **Authorization / control** — deciding whether to *allow* the connection |
 
```python
async def connect(self):
    if self.scope["user"].is_authenticated:
        await self.accept()
    else:
        await self.close()
```
 
### Why Not Use the Default `AuthMiddlewareStack`?
 
Django's built-in `AuthMiddlewareStack` is designed for **session + cookie**-based authentication. Since this flow uses **token-based** authentication instead, a **custom middleware** is required.
 
### Why Tokens Are Needed at All
 
In modern architectures — APIs, mobile apps, Single Page Applications (SPAs) — the server is typically **stateless**: it doesn't store "sessions" (server-side memory of who's logged in). A token acts as portable proof of identity that the client presents with each request.
 
### Important Limitation: No Custom Headers in Browser WebSockets
 
```
headers: { Authorization: "Token abc" }   // ❌ Not possible from a browser WebSocket
```
This is a **browser restriction**. That's exactly why the token is sent via **query parameters** instead.
 
### Security Considerations for Token-in-URL
 
⚠️ Putting a token directly in the URL has downsides:
- It can be **logged** (server access logs).
- It's **visible in browser history**.
**Better alternatives:**
- Use short-lived tokens.
- Send the token in the first WebSocket message instead of the URL.
### JWT Variant (No Database Lookup)
 
Instead of querying the database for the token, decode a JWT directly:
```python
jwt.decode(token, SECRET_KEY)
```
No DB lookup needed — the token itself carries the user's identity.
 
**Summary of this flow:**
1. WebSockets can't use custom headers (in browsers).
2. So the token is sent via query parameters.
3. Custom middleware extracts the token.
4. Middleware finds the corresponding user.
5. Middleware stores the user in `scope["user"]`.
6. The consumer uses that authenticated user for the rest of the connection.

## 21. Who Handles the WebSocket Handshake, and What Is `scope`?
 
### The Pipeline
 
The handshake is handled by the **ASGI server** (e.g., Daphne or Uvicorn), working together with the **Django Channels ASGI application**.
 
**Django Channels** itself is not a single component — it's the entire pipeline of middleware + routing + consumers that sits behind the ASGI server:
 
```
ASGI Server
     ↓
Channels Application (asgi.py — the entry point of Channels)
     ↓
Middleware
     ↓
Router
     ↓
Consumer
```
 
### Step-by-Step: From Raw Request to `connect()`
 
**Step 1 — Browser sends the upgrade request:**
```
GET /ws/chat/?token=abc123 HTTP/1.1
Upgrade: websocket
Connection: Upgrade
```
 
**Step 2 — The ASGI server receives it and builds a `scope`.**
 
A WSGI server (e.g., Gunicorn) converts a raw HTTP request into a Django request object. Similarly, an ASGI server converts the raw connection into a **`scope`** dictionary — metadata describing the connection, analogous to the `request` object in HTTP:
 
```python
scope = {
    "type": "websocket",
    "path": "/ws/chat/",
    "query_string": b"token=abc123",
    "headers": [...],
}
```
 
This is exactly where `scope["query_string"]` is set — done by the **ASGI server**, not by your application code.
 
**Step 3 — The scope enters the Channels application** (defined in `asgi.py`):
```python
application = ProtocolTypeRouter({
    "websocket": TokenAuthMiddleware(
        URLRouter([...])
    )
})
```
 
**Step 4 — Middleware runs first:**
```python
scope["user"] = user  # set by TokenAuthMiddleware after reading query_string
```
 
**Step 5 — `URLRouter` matches the path** (`scope["path"]`) to select the right consumer class, e.g. `ChatConsumer`.
 
**Step 6 — A consumer instance is created:** `ChatConsumer(scope)` — already holding the modified scope (including `scope["user"]`).
 
**Step 7 — `connect()` is called.** At this point, both `scope["query_string"]` and `scope["user"]` already exist and are usable.
 
### Comparison With Django's HTTP Pipeline
 
| HTTP equivalent | WebSocket equivalent |
|---|---|
| `request` object | `scope` |
| middleware | middleware |
| view | consumer |
 
### Key Insight
 
`scope` is created **only once per connection** and passed through the entire pipeline (middleware → router → consumer) — it isn't recreated for each message.

* WebSocket doesn't really have "requests" in this sense — once connected, you just have a stream of messages
 
### Common Mistake
 
❌ *"`connect()` receives the raw request."*
✅ It actually receives the **already-processed `scope`** — after middleware has already run.

## 22. WebSocket Connection Lifecycle in Django Channels (Detailed)
 
### Identity Is Established Once, Not Per Message
 
The server does **not** re-identify the user for every message. Instead:
1. The user is identified **once**, during the initial connection.
2. That identity is stored in the connection's `scope`.
3. **Every subsequent message on that same connection** belongs to that same user — no re-authentication happens.
This works because each WebSocket connection maps to exactly **one consumer instance**:
```
1 client ↔ 1 consumer instance
```
 
With multiple users, each gets a completely separate consumer instance and scope:
```
User1 → Consumer1 (scope.user = user1)
User2 → Consumer2 (scope.user = user2)
```
 
All messages from User1 — msg1, msg2, msg3 — always route to the *same* Consumer1 instance, carrying the *same* `scope.user`.
 
### Full Flow: First Connection, Then Messages
 
**First-time connection:**
1. Client creates the socket: `new WebSocket("ws://localhost:8000/ws?token=abc")`.
2. **HTTP handshake** — the browser sends the upgrade request; the ASGI server creates the `scope`. (Technically, this is not a "plain" HTTP handshake — it's specifically a **WebSocket handshake conducted over HTTP**.)
3. **Middleware** extracts the token, authenticates, and sets `scope["user"]`.
4. **Routing** — `URLRouter` selects the matching consumer.
5. **Consumer instance created** — `consumer = ChatConsumer(scope)`.
6. **`connect()` runs — only once** — `await self.accept()` opens the connection.
**First message from the user:**
7. `socket.send("Hello")` triggers `receive()` (or `receive_json()`) on the **same** consumer instance, with the **same** `scope["user"]`.
 
**Subsequent messages:**
8. Each further `socket.send(...)` call goes straight to `receive()` on the same connection/instance. It does **not** re-run `connect()`, middleware, or routing — that pipeline is **skipped** entirely after the initial handshake.
 

* Http handshake looks like this
  ```
    GET /ws/chat/ HTTP/1.1
    Host: example.com
    Upgrade: websocket
    Connection: Upgrade
    Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
    Sec-WebSocket-Version: 13
  ```
```
socket.send()
   ↓
ASGI server
   ↓ (websocket.receive event)
Channels application
   ↓
Directly to the SAME consumer instance
   ↓
receive()
```
 
### When Does `connect()` Run Again?
 
**Only** if the connection is closed **and** the client reconnects — establishing a brand-new connection (and thus a brand-new scope and consumer instance).
 
### How the Server Sends a Response Back
 
```python
await self.send(text_data="Hi client")
```
 
Internally, Channels turns this into an ASGI event:
```python
await send({
    "type": "websocket.send",
    "text": "Hello Client"
})
```
The ASGI server then converts this event into an actual WebSocket frame and sends it over the network.
 
On the browser side, the incoming frame triggers the `onmessage` event with a `MessageEvent` object (a browser-native object — *not* the same as the ASGI `"websocket.send"` event happend in the server):
```javascript
socket.onmessage = (event) => {
  console.log(event.data);
};
```
The browser never sees the raw ASGI event type — it only understands "data arrived," which triggers `onmessage`.The data is 'MessageEvent' object.

### Full Reverse Flow (Server → Client)
 
```
BACKEND:
self.send()
   ↓
Channels creates "websocket.send" ASGI event
   ↓
ASGI server converts it into a WebSocket frame, sent over the network
-----------------------------------------------------
FRONTEND:
Browser receives the frame
   ↓
Extracts the data
   ↓
Creates a MessageEvent object
   ↓
Calls onmessage(event)

```
### Connection Close
 
1. Either side calls `socket.close()`.
2. ASGI sends a disconnect event: `{"type": "websocket.disconnect"}`.
3. Channels calls `async def disconnect(self, code): ...` on the consumer.
4. The consumer instance is destroyed.

*  Both the client and the server are equally capable of initiating the close, and the cleanup process (disconnect()) needs to happen regardless of who started it from server.
case-A
```
// Frontend code
socket.close();
```
case-B 
```
# Backend code (Consumer)
async def receive(self, text_data):
    if invalid_token:
        await self.close(code=4001)  # server decides to close it

```

### Event-to-Method Mapping (Complete Reference)
 
| Frontend action | ASGI event | Consumer method |
|---|---|---|
| `new WebSocket()` | `websocket.connect` | `connect()` |
| `socket.send()` | `websocket.receive` | `receive()` |
| `socket.close()` | `websocket.disconnect` | `disconnect()` |
 
### Full Lifecycle Diagram
 
```
FRONTEND:
new WebSocket()
   ↓
HTTP Upgrade Request
   ↓
ASGI: create scope, create websocket.connect event
   ↓
CHANNELS: middleware (auth) → routing → create consumer instance
   ↓
connect() runs
   ↓
connection open
-----------------------------------------------------
MESSAGES:
socket.send() → ASGI (websocket.receive) → Channels → receive()
(repeats many times)
-----------------------------------------------------
RESPONSE:
self.send() → frontend onmessage
-----------------------------------------------------
CLOSE:
socket.close() → ASGI (websocket.disconnect) → disconnect() → instance destroyed
```
 
### Final Understanding
 
```
connect()    → runs once
receive()    → runs many times
disconnect() → runs once
```
 
**One-line summary:** WebSocket = connect once → communicate many times → disconnect once.

* Both sides ultimately deal with raw WebSocket frames over the network. But each side wraps those frames in its own local representation — the server uses ASGI events, the browser uses event objects (MessageEvent, etc.) — neither of which travels over the network itself.

**server to browser**
```
SERVER SIDE                    NETWORK                    BROWSER SIDE
ASGI event                                                MessageEvent
(Python dict)                                              (JS object)
     ↓                                                          ↑
     │  ←── these are LOCAL representations, never sent ──→     │
     ↓                                                          ↑
  converted to a                                          constructed from
  WebSocket FRAME    ────────────►  raw bytes  ────────────►   a WebSocket FRAME
  (bytes on the wire)                                       (bytes on the wire)

```
**browser to server**
```
BROWSER SIDE                   NETWORK                    SERVER SIDE
MessageEvent / Send
(JS object/string)                                              
     │                                                          │
     ▼                                                          │
  converted to a                                                │
  WebSocket FRAME     ───────────►  raw bytes  ───────────►  interpreted by ASGI 
  (bytes on the wire)                                       as an ASGI event
                                                            (Python dict)
```

## 23. Browser WebSocket Events Recap (Mapped to Backend)
 
| Browser-side | What it does | Backend equivalent (Django Channels) |
|---|---|---|
| `new WebSocket()` | Opens the connection (handshake) | `connect()` |
| `socket.send()` | Sends a message to the server | `receive()` |
| `socket.onmessage` | Fires when the server sends data | Triggered by `self.send()` on the backend |
| `socket.onclose` | Fires when the connection closes | `disconnect()` |
 
```javascript
const socket = new WebSocket("ws://localhost:8000/ws");
 
socket.onopen = () => {
  console.log("Connected");
  socket.send("Hello Server");
};
 
socket.onmessage = (event) => {
  console.log("Server says:", event.data);
};
 
socket.onclose = () => {
  console.log("Disconnected");
};
```
 
**Important rule:** You can only send messages **after** `onopen` has fired.
 
You never call `receive()` manually on the backend — Channels (or FastAPI's loop) triggers it automatically whenever a message arrives.

## 24. Plain Python WebSocket Chat App (No Framework)
 
### What's Being Built
 
A simple real-time chat room, without Django or Flask, using:
- `asyncio` — for handling many concurrent users.
- `websockets` (Python library) — for the WebSocket protocol itself.
### Architecture
 
```
Browser → WebSocket → Python Server → Broadcast → Other Browsers
```
 
Two parts: a Python server (backend) and a browser client (frontend).
 
### Server-Side Logic
 
**1. Track connected clients:**
```python
clients = set()
```
 
**2. Handle each connection:**
```python
async def handle(ws):
    clients.add(ws)
    try:
        async for msg in ws:## ← this LOOP keeps running, again and again
            #this block runs EVERY TIME a new message arrives on this connection
            await asyncio.gather(*[
                c.send(msg) for c in clients if c != ws
            ])
    finally:
        clients.remove(ws)
```
* ws is the WebSocket connection object itself — think of it as a live, ongoing pipe between this one specific user and the server. It is not a single message, and it doesn't "contain" messages the way a list contains items.
* ws is something called an async iterable — an object that you can "loop over," but instead of looping over items already sitting in memory (like a list), each "next item" only becomes available when it actually arrives over the network.

|Normal `for` loop|`async for` loop over a WebSocket|
|`for x in [1, 2, 3]:` — all 3 items already exist, loop runs through them instantly|`async for msg in ws:` — there's no pre-existing list of messages; each iteration waits until the next message actually arrives from the network|

| Line | Purpose |
|---|---|
| `clients = set()` | Tracks all currently connected users |
| `async def handle(ws)` | Runs once per connecting user; `ws` is that user's connection |
| `clients.add(ws)` | Registers the user as active |
| `async for msg in ws` | Listens forever for incoming messages |
| `asyncio.gather(*[c.send(msg) for c in clients if c != ws])` | **Broadcasts** the message to every other connected client (excludes the sender) |
| `clients.remove(ws)` | Cleans up when the user disconnects |
 
* `handle(ws)` is called once per connection — but that one call doesn't finish quickly. It stays alive, running, for the entire duration of that connection, means until it closes the connection.
* So within one single call to `handle(ws)`, the async for loop is what repeats the broadcast block — every time a new message shows up on that connection, the loop body runs again, without `handle()` itself being called again.

**3. Start the server:**
```python
websockets.serve(handle, "localhost", 6789)
await asyncio.Future()  # keeps the server running forever
```
This starts a WebSocket server at `ws://localhost:6789`, where `handle` runs for every new connection, and `await asyncio.Future()` is a simple trick to keep the event loop running indefinitely.

* `await asyncio.Future()`This line has nothing to do with creating WebSocket objects. Its only job is to keep the program from exiting. websockets.serve(...) sets up the listening server and returns almost immediately — without something to keep the program alive, your script would just end right after starting the server. asyncio.Future() creates a "Future" that never resolves, so await-ing it just pauses forever, keeping the whole event loop (and thus the server) running indefinitely.

**NOte it**
1. "How long does this ONE user's connection stay being handled?" → answered by async for msg in ws
2. "How does the overall Python program keep running at all, so it can accept new connections in the first place?" → answered by await asyncio.Future() 

### Client-Side (Browser HTML + JS)
 
```javascript
const ws = new WebSocket("ws://localhost:6789");
const c = document.getElementById("chat");
const txt = document.getElementById("messageInput").value;
ws.onmessage = e => c.value += e.data + "\n";   // receive and display others messages
 
ws.send(txt);                                    // send this user's message
c.value += "<You>: " + txt + "\n";                // show own message instantly
document.getElementById("messageInput").value = "";  // clear input after sending
```
### Full Flow (Two Users)
 
1. User A sends a message: `ws.send("Hello")`.
2. The server receives it.
3. The server loops through all connected clients.
4. It sends the message to B, C, D, etc. (everyone except A).
5. B receives it via `onmessage` and displays it.
This pattern is called **broadcasting**.
 
### Why This Approach (async + WebSocket)
 
Compared to older socket/threading approaches, this is:
- Easier to scale to many users.
- Free of complex manual threading.
- Naturally web-friendly.
- Less code overall.
### Key Concepts
 
1. **WebSocket** — full-duplex (two-way) communication without repeated HTTP requests.
2. **`asyncio`** — runs multiple tasks concurrently without needing separate threads.
3. **Event-driven** — code automatically runs in response to events (message arrives, connection opens/closes).
### Running It
 
```bash
pip install websockets
python server.py
```
Then open `client.html` in multiple browser tabs to simulate multiple users.

- Viewing message history.

## 26. Quick-Reference Glossary
 
| Term | Meaning |
|---|---|
| **WebSocket protocol** | The network-level protocol enabling persistent, full-duplex communication over a single TCP connection |
| **WebSocket API** | The JavaScript interface browsers expose for using the WebSocket protocol |
| **Full-duplex** | Both sides can send and receive data simultaneously over the same connection |
| **Handshake / Upgrade** | The initial HTTP request (`Upgrade: websocket`) that switches a connection from HTTP to WebSocket (`101 Switching Protocols`) |
| **`ws://` / `wss://`** | URL schemes for WebSocket (unencrypted / encrypted, paralleling `http://` / `https://`) |
| **ASGI** | Asynchronous Server Gateway Interface — the async counterpart to WSGI, required for WebSocket support in Django |
| **WSGI** | Web Server Gateway Interface — Django's traditional synchronous, HTTP-only interface |
| **Consumer** | In Django Channels, the WebSocket equivalent of a Django view |
| **`scope`** | Metadata dictionary describing a WebSocket connection (path, headers, query string, user, etc.), created once by the ASGI server and passed through the pipeline |
| **Daphne** | An ASGI server commonly used to run Django Channels applications |
| **Sticky session** | A load-balancer configuration that always routes a given client to the same backend server — often needed for stateful WebSocket connections |
| **JWT (JSON Web Token)** | A self-contained, signed token encoding user identity/claims, verifiable without a database lookup |
| **`HttpOnly` cookie flag** | Prevents JavaScript from reading the cookie, protecting against XSS-based token theft |
| **`SameSite` cookie flag** | Controls whether a cookie is sent on cross-site requests, helping prevent CSRF |
| **CSRF** | Cross-Site Request Forgery — an attack exploiting automatically-sent cookies to trigger unwanted actions |
| **XSS** | Cross-Site Scripting — an attack injecting malicious JavaScript into a page |

* **What "sticky session" actually does**
    * A sticky session is a load balancer setting that says: "once a specific client first connects to Server1, always send that SAME client back to Server1 for all future connections/requests — never switch them to a different server."
```
    Users
    ↓
Load Balancer
↙    ↓    ↘
Server1 Server2 Server3
```
```
User A's FIRST connection → Load balancer picks Server1
        ↓
Load balancer "remembers": User A → Server1 (sticky)
        ↓
User A reconnects later → Load balancer checks its memory:
        "Oh, User A always goes to Server1" → routes to Server1 again
```
