## 1. What Is the WebSocket API?

**WebSocket API** is a browser-side JavaScript API used to open a **persistent connection** between a client (e.g., a browser) and a server, so both sides can send data to each other without repeatedly making new HTTP requests.
- It is **bidirectional** and **full-duplex** ÔÇö data can move both ways at the same time, over a **single connection**.It enables true **real time communication**.
- Once the connection is established, the server can push new data to the browser immediately, and the browser can send data back whenever it wants.

> **Important distinction:** The **WebSocket API** is not the same thing as the **WebSocket protocol**.
> - The **protocol** defines *how* client and server communicate at the network level.
> - The **API** is the JavaScript interface the browser provides for using that protocol (`WebSocket`, `onopen`, `onmessage`, etc.).

### Difference between websocket and webRTC**
* WebSocket is built for Client-to-Server communication (like a browser talking to a backend database).

* WebRTC (Web Real-Time Communication) is built for Peer-to-Peer (P2P) communication (like a browser talking directly to another browser).
* WebRTC is not a single protocol ÔÇö it's a framework/set of standards that bundles together multiple existing protocols and APIs to enable real-time peer-to-peer communication.

### Why WebSocket Was Created
 
Before WebSocket, real-time-ish behavior was faked using:
- **Short polling** ÔÇö the client repeatedly asking the server for updates.
- **Long polling** ÔÇö  the client sends a request and the server purposefully holds the connection open, server responding only when new data becomes available, after which the client must immediately send a brand new request to repeat the process(reuest- response cycle).

**Problems with both:** too many requests, high server load, and delays.
 
WebSocket solves this by keeping a single connection open and letting the server send data instantly, without the client asking again and again.

### When to Use It
 
Use the WebSocket API when the application needs **instant, two-way communication** between browser and server. Typical use cases:
 
- Chat systems
- Video or live interaction apps
- Multiplayer games
- Collaborative editing (shared documents)
- Live dashboards and visualizations
- Event update systems


### When NOT to Use WebSocket
 
Avoid WebSocket when:
- Data is static
- Requests happen rarely
Keeping a connection open in these cases just wastes server resources.

### Video calling in browser - WebRTC and WebSocket
The two pieces
1. WebRTC ÔÇö carries the actual audio/video

    * Once two browsers are connected, the actual video and audio stream flows directly between them (peer-to-peer), not through your server.
    * Runs mostly over UDP, optimized for low latency ÔÇö a bit of dropped video data is fine; nobody wants delay.
    * Handles the media capture (camera/mic), encoding, and the direct peer connection.

2. WebSocket ÔÇö handles signaling (the "let's connect" handshake)

* Before two browsers can talk directly via WebRTC, they need to exchange some setup information: "here's my video format," "here's my network address," "I want to call you."
* This exchange is called signaling, and WebRTC does not define how to do it ÔÇö it's left up to the developer.
* WebSocket is a common (but not the only) way to implement signaling, because it's a persistent connection that lets your server relay these setup messages between the two browsers in real time.
 
 ## 2. HTTP vs WebSocket
 
### What Is HTTP?
 
HTTP is a **request Ôćĺ response protocol**.
 
```
Client Ôćĺ Request Ôćĺ Server
Client ÔćÉ Response ÔćÉ Server
ÔŁî Connection usually ends
```
 
**Key ideas:**
- Client always starts the exchange.
- Server only replies.
- No continuous communication.
- Server **cannot push data** to the client without a prior request (without tricks like polling).
This is why HTTP is called **unidirectional** (one-way per request).

### What Is WebSocket?
 
WebSocket is a **persistent, two-way** communication channel.
 
```
Client Ôćö Server (both can send anytime)
Ôťů Connection stays open
```
 
**Key ideas:**
- After the connection opens, both sides can talk freely.
- No need to send a new request every time.
This is why WebSocket is called **full-duplex**.


### Core Differences
 
| Aspect | HTTP | WebSocket |
|---|---|---|
| **Communication style** | Client Ôćĺ Server only; server can't push without a request | Client Ôćö Server, both directions, anytime |
| **Connection behavior** | Opens Ôćĺ responds Ôćĺ closes | Opens once Ôćĺ stays open |
| **Overhead** | Sends full headers every request (heavy) | Small frames after handshake (lightweight) |
| **Latency** | Slower ÔÇö new request/response cycle each time | Faster ÔÇö no repeated setup |
| **Scalability** | Easier (stateless) ÔÇö supports vertical & horizontal scaling well | Harder (stateful) ÔÇö needs sticky sessions or shared state (e.g., Redis) for horizontal scaling |
| **Typical use cases** | Static pages, normal APIs, file downloads, one-time data | Chat apps, live notifications, online games, stock updates |

### Relationship Between HTTP and WebSocket
 
WebSocket connections **start as HTTP** and then **upgrade**:
 
```
HTTP request Ôćĺ Upgrade Ôćĺ WebSocket connection
```
 
So they are different protocols, but WebSocket **uses HTTP to start** the connection. They are not competitors ÔÇö WebSocket is **not a replacement** for HTTP/REST; modern apps often use both together (REST for normal data, WebSocket for real-time features).
### One-Line Summary
 
**HTTP = ask and get once.**
**WebSocket = stay connected and keep talking.**

## 3. Does WebSocket Use TCP?
 
**Yes ÔÇö WebSocket runs on top of TCP, not UDP.**
 
### How It Works
 
**1. Starts with HTTP (over TCP):**
 
Client sends:
```
Upgrade: websocket
```
 
Server responds:
```
HTTP/1.1 101 Switching Protocols
```
 
This handshake happens entirely over **TCP**.
 
**2. After the upgrade:**
- The connection becomes a **WebSocket** connection.
- It still uses the **same underlying TCP connection** ÔÇö no new connection is opened.


```
WebSocket = HTTP upgrade + persistent TCP connection
```

### Why TCP?
 
WebSocket needs:
- Ôťů Reliable delivery
- Ôťů Ordered messages
- Ôťů No data loss
TCP provides exactly this.

### Ports

| `wss://` (secure WebSocket) | 443 |
 
WebSocket runs **on top of existing HTTP infrastructure**, which is why it reuses HTTP's default ports.

## 4. Is WebSocket Only Used on the Web?
 
**No ÔÇö WebSocket is a protocol, not just a browser feature.**
 
It's a common misunderstanding that WebSockets are web-only, because:
- They started in browsers (JavaScript).
- They're heavily used in web apps (chat, live updates).
**Reality:** any application that can open a TCP connection and speak the WebSocket protocol can use WebSockets.
 
### Where WebSockets Are Used
 
- **Web browsers** ÔÇö chat apps, notifications, live dashboards.
- **Mobile apps (Android/iOS)** ÔÇö real-time messaging, ride-tracking apps (e.g., Uber-style live location).
- **Backend services** ÔÇö microservice communication, real-time data pipelines. Example libraries: Python (`asyncio`, `websockets`), Node.js (`ws`, `socket.io`).
- **Games** ÔÇö multiplayer games, live player position updates.
- **IoT devices** ÔÇö smart devices and sensors streaming live data.

### WebSocket's framing overhead compared to HTTP
 
WebSocket reduces overhead  by exchanging mostly just message data ÔÇö not exactly "only" message data, but very close to it, with minimal extra framing overhead compared to HTTP's full headers on every request fpor message data.
* Http has more overhead for sending message data.

### Can WebSocket carry audio/video?
**Technically yes, but it's not what it's designed or used for.**
WebSocket can send two kinds of data:

* Text frames (usually UTF-8 / JSON)
* Binary frames (raw bytes ÔÇö could be an image, a chunk of audio, anything)

So in theory, you could chop up audio/video into binary chunks and send them frame-by-frame over WebSocket. Some applications do use WebSocket for low-volume audio/video-adjacent tasks (like sending periodic snapshots, or low-bitrate audio clips).

**But for real-time, continuous audio/video calls, WebSocket is a poor fit for audio/video**,becasue
|Issue|Why it hurts video/audio|
|Runs over TCP|TCP guarantees delivery and order ÔÇö so if one packet is lost, it waits and retransmits before delivering anything newer. For live video, a late frame is useless; you'd rather drop it and move on.|
|No built-in media optimization|No native support for things like jitter buffering, codec negotiation, or adaptive bitrate|
|Goes through a server|All data is relayed via your server ÔÇö adds latency and bandwidth cost compared to direct peer-to-peer|

That's exactly why WebRTC exists ÔÇö it uses UDP (via SRTP), which tolerates dropped packets gracefully (a skipped video frame is fine ÔÇö better than freezing to wait for it), and it connects peer-to-peer, skipping the server entirely for media.

## 5. REST API vs WebSocket API
 
### Core Models
 
- **REST API** ÔÇö uses the HTTP **requestÔÇôresponse model**: client sends a request, server sends a response, connection ends.
- **WebSocket API** ÔÇö uses a **persistent, full-duplex connection**: one connection stays open, and both client and server can send data anytime.
### Communication Style
 
```
REST:      Client Ôćĺ Request Ôćĺ Server Ôćĺ Response Ôćĺ Done   (one-time communication)
WebSocket: Client Ôćö Server (continuous)                    (real-time communication)
```

### Detailed Comparison
 
| Feature | REST API | WebSocket API |
|---|---|---|
| **State** | Stateless (no stored data) | Stateful (stores connection data) |
| **Direction** | Unidirectional (either server or client per request) | Bidirectional |
| **Model** | RequestÔÇôresponse | Full-duplex |
| **Headers/Overhead** | HTTP headers sent with every request | No repeated header overhead once connected |
| **Connection** | New TCP connection per request | Single persistent connection |
| **What identifies a "request"** | HTTP Method + path, fresh each time | The connection itself, identified by client IP/port, stays the same for the whole session |
| **Speed** | Slower | Faster |
| **Buffering** | No memory/buffer needed to hold data | Needs memory/buffers to manage live data |
| **Scaling** | Supports vertical and horizontal scaling easily | Mainly easy for vertical scaling; horizontal scaling is harder(bcz of stateful) |

### Advantages and Disadvantages
 
**REST API**
- Ôťů Advantages: Simplicity, flexibility, caching, statelessness.
- ÔŁî Disadvantages: Limited real-time capability; performance overhead since each request/response must carry all necessary info; can get complex for large apps.
**WebSocket API**
- Ôťů Advantages: Real-time communication, efficient (one persistent bidirectional connection), scalable (one server can support thousands of websocket connections, bcz of websocket is lightweight), low overhead (single connection for ongoing transmission).
- ÔŁî Disadvantages: More complex to design and implement than REST; not supported in all environments; vulnerable to security issues if not properly secured.
**Similarities:** Both are used for web applications between client and server, can be customized, and can be secured using authentication and encryption.


### Use-Case Summary
 
| Best for REST | Best for WebSocket |
|---|---|
| Fetching data occasionally | Real-time apps |
| CRUD operations (GET, POST, PUT, DELETE) | Continuous updates |
| User data, product lists | Chat apps, live stock prices, online games |
 
### Simple Analogy
 
- **REST** Ôćĺ like sending an email ­čôž: send Ôćĺ wait Ôćĺ receive Ôćĺ done.
- **WebSocket** Ôćĺ like a phone call ­čô×: stay connected, talk anytime.

### Important Insight
 
WebSocket is **not a replacement** for REST:
- REST = simple, scalable.
- WebSocket = fast, real-time.
Modern applications often use **both together** ÔÇö REST for standard data operations and WebSocket for real-time features.
 
**Final answer:** REST API uses a requestÔÇôresponse model over HTTP, where each interaction is independent, while WebSocket API maintains a persistent connection allowing real-time, two-way communication between client and server.

### Vertical Scaling
 
Increasing the power of **one server** (more CPU, more RAM):
```
1 server Ôćĺ stronger server
```
 
### Horizontal Scaling
 
Adding **more servers**, distributing load through a **load balancer**.
 
```
        Users
          Ôćô
    Load Balancer
     ÔćÖ    Ôćô    Ôćś

```
 
**Example:** 1,000 users on 1 server would overload it. Spread across 5 servers, each handles ~200 users ÔÇö faster and more stable.
 
**Common load balancers:** Nginx, HAProxy, AWS ELB/ALB.
 
### Scaling by Protocol
 
| | HTTP | WebSocket |

## 7. WebSocket vs WebRTC
 
### Core Idea
 
- **WebSocket** ÔÇö **client Ôćö server** communication. Goes through a server, persistent connection, used for general real-time apps.
- **WebRTC** ÔÇö **peer-to-peer (P2P)** communication. Direct **client Ôćö client**; the server is only used for initial setup (signaling). Designed for media (audio/video).
### Deep Comparison
 
| Aspect | WebSocket | WebRTC |
|---|---|---|
| **Architecture** | Client Ôćö Server Ôćö Client (all data goes through the server) | Client Ôćö Client (direct; server not used after setup) |
| **Protocol** | TCP | UDP (mostly) |
| **Latency** | Low | Ultra-low (no middle server, uses UDP) |
| **Data type** | Any ÔÇö text, JSON, binary | Optimized for audio, video, live streaming |
| **Reliability** | Reliable (TCP guarantees delivery) | May drop packets (UDP) ÔÇö acceptable for live video/audio |
| **Scalability(server load grows as more users join)** | Server handles all connections; more users = more server load (one persistent connection per client) | Less server load (P2P), but hard to scale to large groups since many direct client-to-client connections are needed |
| **Setup complexity** | Simple | Complex ÔÇö needs ICE candidates, STUN/TURN servers |

### Common Misconception: "WebRTC Is Connectionless Because It Uses UDP"
 
**Not correct.**
 
- **UDP is connectionless.**
- **WebRTC is *not* connectionless** ÔÇö even though it uses UDP, it builds a connection-like system on top of it.
WebRTC adds, on top of raw UDP:
- Connection setup (**ICE**)
- Session management
- Encryption (**DTLS**)
- Optional reliability for data channels
WebRTC still needs servers for **connection setup (signaling)** and **NAT traversal (STUN/TURN)**. After that, communication is direct.
 
### When to Use What
 
| Use WebSocket for | Use WebRTC for |
|---|---|
| Chat apps ­čĺČ | Video calls ­čô╣ |
| Notifications ­čöö | Voice calls ­čÄž |
| Real-time dashboards ­čôŐ | Screen sharing ­čľą´ŞĆ |
 
### Quick Summary Table
 
| | WebSocket | WebRTC |
|---|---|---|
| Communication | ClientÔÇôServer | Peer-to-Peer |
| Protocol | TCP | UDP |
| Latency | Low | Very low |
| Best for | Data | Audio/Video |
| Complexity | Easy | Complex |
 
**Final answer:** WebSocket is used for real-time communication between client and server over TCP, while WebRTC enables direct peer-to-peer communication (usually over UDP), making it ideal for ultra-low-latency applications like video and voice calls.

## 8. WebSocket Connection Lifecycle (Handshake)
 
### Step-by-Step Flow
 
1. **Client sends an HTTP request** requesting an upgrade:
```
   GET /ws HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade
```
   This is **not** for sending data ÔÇö it's asking, "Can we switch to WebSocket?"
 
2. **Server responds:**
```
   HTTP/1.1 101 Switching Protocols
```
   This means "Yes, switching to WebSocket."
 
3. **The connection becomes persistent.** HTTP is no longer used for messages ÔÇö both sides can now send data anytime, with no more requestÔÇôresponse cycles.
### Why `ws://` Instead of `http://`?
 
Even though the handshake itself is an HTTP request, using `ws://` tells the browser: *"Start a WebSocket connection (not a normal HTTP request)."* This matters because the browser needs to:
- Add the special `Upgrade` and `Connection` headers.
- Keep the connection open.
- Switch into the WebSocket protocol after the handshake completes.
This only happens automatically when using the `WebSocket()` constructor with a `ws://` or `wss://` URL.
 
### Stateless vs Stateful
 
- **HTTP** is **stateless** ÔÇö each request is handled independently.
- **WebSocket** is **stateful** ÔÇö the connection between client and server stays alive until either side terminates it.

## 9. Main WebSocket Events (Browser API)
 
When you create a connection:
 
```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws");
```
 
You can listen for these core lifecycle events in Browser API:
 
| Event | Fires when... | Notes |
|---|---|---|
| **`open`** | Connection is successfully established (after handshake) | Start of communication |
| **`message`** ÔşÉ | Server sends data to the client | Most-used event ÔÇö this is how you receive updates |
| **`close`** | Connection is closed | Provides `event.code` (why it closed) and `event.reason` (optional message) |
| **`error`** | Something goes wrong (server unreachable, network issue, protocol error) | Useful for debugging |
 
### Two Equivalent Syntaxes
 
```javascript
// Using addEventListener
socket.addEventListener("open", () => { console.log("Connected"); });
socket.addEventListener("message", (event) => { console.log(event.data); });
 
// Using direct properties (same behavior)
socket.onopen = () => { console.log("Connected"); };
socket.onmessage = (event) => { console.log(event.data); };
```
 
**Summary:**
- `open` Ôćĺ connection ready, start sending.
- `message` Ôćĺ main application logic (receiving data).
- `close` Ôćĺ cleanup or reconnect logic.
- `error` Ôćĺ debugging.
> **Rule:** You can only call `socket.send()` *after* the `open` event has fired.

## 10. WebSocket in FastAPI
 
### Setting Up an Endpoint
 
FastAPI provides a dedicated decorator for WebSocket routes:
 
```python
@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    while True:
        data = await websocket.receive_text()
        await websocket.send_text(data)
```
 
- `@app.websocket("/ws")` defines a **WebSocket endpoint**, not a normal HTTP API route.
- `await websocket.accept()` is the most important line ÔÇö it accepts the connection. Without it, the connection fails.
- The `while True` loop keeps the connection alive: wait for a message Ôćĺ receive it Ôćĺ send a response.
### Matching Frontend Code
 
```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws");
 
socket.addEventListener("open", () => {
  console.log("Connected");
});
 
socket.send("Hello");
 
socket.onmessage = (event) => {
  console.log(event.data);
};
```
 
### Visual Flow
 
```
Browser                                FastAPI Server
   |                                          |
   |---- connect (/ws) -------------------->  |  new WebSocket('ws://127.0.0.1:8000/ws')
   |<--- accept() --------------------------  |  websocket.accept() ÔÇö connection established
   |                                          |
   |---- "Hello" --------------------------->  |
   |<--- "Hello" ----------------------------  |
   |                                          |
   |---- "Hi" -------------------------------> |
   |<--- "Hi" --------------------------------  |
```
 
The same connection is reused for every message ÔÇö no new HTTP requests are made.
 
### Why `async`/`await` Matters
 
WebSocket communication needs continuous waiting and sending. FastAPI uses `async`/`await` so the server is **non-blocking** ÔÇö it can handle multiple users concurrently without separate threads per connection.
 
### Real-World Use Cases
 
Chat apps, live notifications, stock price updates, multiplayer games ÔÇö anywhere the server needs to push data without waiting for a request.

## 11. FastAPI Chat App Example (Multi-User Broadcast)
 
### What This Project Does
 
A real-time chat app where:
- Multiple users connect.
- Each user sends messages.
- The server distributes (broadcasts) messages to other connected users.
### High-Level Architecture
 
```
Browser (User A) ÔöÇÔöÉ
                   ÔöťÔöÇÔöÇ> FastAPI Server ÔöÇÔöÇÔöÇ> All Users
Browser (User B) ÔöÇÔöś
```
 
The server acts as a **middleman** that receives and distributes messages.
 
### Frontend Flow
 
```javascript
// 1. Create connection with a unique user ID
const socket = new WebSocket('ws://127.0.0.1:8000/ws/' + id);
 
// 2. On connect
socket.addEventListener('open', () => {
  showMessage('Connected to server.');
});
 
// 3. Sending a message
socket.send(inputValue);
 
// 4. Receiving a message
socket.onmessage = (event) => {
  showMessage(event.data);
};
```
 
### Backend: `ConnectionManager` Class
 
This class tracks all connected clients:
 
```python
class ConnectionManager:
    def __init__(self):
        self.active_connections: List[WebSocket] = []
 
    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)
 
    def disconnect(self, websocket: WebSocket):
        self.active_connections.remove(websocket)
 
    async def send_personal_message(self, message: str, websocket: WebSocket):
        await websocket.send_text(message)
 
    async def broadcast(self, message: str, websocket: WebSocket):
        for connection in self.active_connections:
            if connection == websocket:
                continue
            await connection.send_text(message)
```
 
| Method | Purpose |
|---|---|
| `connect()` | Accepts the connection and adds the user to the active list |
| `disconnect()` | Removes a user when they leave |
| `send_personal_message()` | Sends a message to one specific user |
| `broadcast()` | Sends a message to all *other* connected users (skips the sender, since the sender already got their own copy) |
 
### WebSocket Endpoint
 
```python
connectionmanager = ConnectionManager()
@app.websocket("/ws/{client_id}")
async def websocket_endpoint(websocket: WebSocket, client_id: int):
    await connectionmanager.connect(websocket)
    try:
        while True:
            data = await websocket.receive_text()
            # Send back to the same user
            await connectionmanager.send_personal_message(f"You : {data}", websocket)
            # Broadcast to everyone else
            await connectionmanager.broadcast(f"Client #{client_id}: {data}", websocket)
    except WebSocketDisconnect:
        connectionmanager.disconnect(websocket)
        await connectionmanager.broadcast(f"Client #{client_id} left", websocket)
```
 
- `WebSocketDisconnect` is the exception raised when a client disconnects (tab closed, network lost, etc.). On disconnect, the user is removed from the connection list and others are notified.
### Full Flow
 

| Step | What happens | Method/event |
|---|---|---|
| 1. User opens the page `ws://127.0.0.1:8000/ws/` | Browser creates the WebSocket and starts the handshake , and the hanshake request (HTTP upgarde) is send to server | `new WebSocket(...)` triggers it |
| 2. Server accepts it | Backend explicitly accepts the connection in hadshake | `await websocket.accept()` |
| 3. Handshake completes | Browser confirms the connection is now open | `socket.onopen` fires (frontend) |
| 4. User sends a message | Frontend sends data over the open socket | `socket.send(...)` |
| 5. Server receives it | Backend's loop picks up the incoming text | `data = await websocket.receive_text()` |
| 6. Server sends it back to sender + broadcasts to others | Two separate sends from the backend | `send_personal_message()` Ôćĺ `websocket.send_text(...)`; `broadcast()` Ôćĺ `connection.send_text(...)` for each other client |
| 7. Repeat indefinitely | The `while True` loop keeps listening for more messages | ÔÇö |
### Key Concepts
 
- **Persistent connection** ÔÇö one connection handles many messages; no repeated HTTP requests.
- **Connection list** (`self.active_connections`) ÔÇö how the server tracks who's currently connected.
- **Broadcast logic** ÔÇö the core mechanism behind chat apps.
- **Infinite loop** (`while True`) ÔÇö keeps the connection alive.
### Limitation of example and Real-World Improvement
 
This example stores `active_connections` **in memory**, which means:
- ÔŁî Not scalable ÔÇö it only works on a single server instance.
Real-world apps typically use **Redis**, message queues, or pub/sub systems to share connection state across multiple server instances.
 
**One-line summary:** FastAPI chat app = WebSocket connections + a connection list + broadcast messages.

| **Your code** | Writes `new WebSocket(...)` ÔÇö this is *your* instruction, executed by the browser's JS engine |
| **Browser (again, after your call)** | Once your code creates the object, the browser's internals take over: it sends the actual handshake request over the network, manages the TCP connection, parses incoming frames, and fires events back to your code (`onopen`, `onmessage`, etc.) |

## 12. WebSocket with Django (Django Channels)
 
### The Problem
 
Plain Django (HTTP) follows: client Ôćĺ request Ôćĺ server Ôćĺ response Ôćĺ connection closed. The server **cannot send data anytime** ÔÇö it can only reply when asked.
 
With WebSockets: client Ôçä server, connection stays open, enabling real-time communication where the server can push data anytime. Useful for chat apps, live notifications, dashboards.
 
### Why Plain Django Can't Do WebSockets
 
Django, by default, uses **WSGI** (synchronous) ÔÇö which only supports HTTP. To add WebSocket support, Django needs **Django Channels**, which switches to **ASGI** (asynchronous).
 
### Architecture
 
```

       Ôćô

                        which loads the `application` object from asgi.py
       Ôćô

       Ôćô

```

**1. Install packages**
```bash

```
- `channels` Ôćĺ adds WebSocket support.
- `daphne` Ôćĺ an ASGI server (like Gunicorn, but async-capable).

    ...,

This tells Django to use ASGI instead of WSGI.
 
**3. ASGI configuration (`asgi.py`)** ÔÇö the core routing decision-maker:

## 13. What Is Daphne?
 
**Daphne is an ASGI server** ÔÇö it runs your Django app and knows how to handle WebSockets, HTTP, and async connections. It's the program that sits between the client and your Django (ASGI) application.
 
### Why Daphne Is Needed
 
**Normal Django (WSGI):**
```
Browser Ôćĺ Gunicorn Ôćĺ Django (WSGI) Ôćĺ Response
```
ÔŁî Only HTTP, no WebSockets.
 
**With Channels + Daphne (ASGI):**
```
Browser Ôćĺ Daphne Ôćĺ Django (ASGI) Ôćĺ Consumer/View
```
Ôťů Handles HTTP, WebSockets, and async tasks.
 
### What Daphne Does
 
1. **Accepts the connection** ÔÇö either HTTP or WebSocket.
2. **Understands the protocol type:**
   | Protocol | Routed to |
   |---|---|
   | HTTP | Django views |
   | WebSocket | Consumer |
3. **Keeps the connection open for WebSockets** ÔÇö the key difference from a traditional WSGI server:
   | Gunicorn | Daphne |
   |---|---|
   | Request Ôćĺ Response Ôćĺ Close | Connect Ôćĺ Stay open Ôćĺ Send/Receive anytime (onlyu for websocket)|

### `ProtocolTypeRouter` ÔÇö The Decision-Maker
 
Inside `asgi.py`:
```python

**HTTP request:**
```

Ôćĺ Routed to `"http"`.

The special `Upgrade: websocket` header signals Ôćĺ routed to `"websocket"`.

## 14. FastAPI vs Django Channels: Receiving Messages
 
### Django Channels (Event-Driven)
 
```python
def receive(self, text_data):
    data = json.loads(text_data)
    message = data['message']
```
- `receive()` is **automatically called** by the framework whenever a message arrives.
- `text_data` is passed in for you; you manually parse the JSON.
- Mental model: *"When a message comes in Ôćĺ run this function."*
### FastAPI (Manual Async Loop)
 
```python
data = await websocket.receive_text()
```
- You **explicitly call** `receive_text()`.
- It asynchronously waits for a message and returns the text directly.
- Mental model: *"Keep listening and fetch messages yourself."*
### Core Difference
 
| | Django Channels | FastAPI |
|---|---|---|
| **Style** | Event-based | Async loop |
| **Receiving data** | Framework auto-calls `receive()` | You call `receive_text()` |
| **Control** | Framework-driven | Developer-driven |
 
### Why the Difference Exists
 
- **Django Channels** is built on an event system that handles the lifecycle automatically: `connect()`, `receive()`, `disconnect()`.
- **FastAPI** is built on explicit `async`/`await` control ÔÇö you manage the loop, receiving, and sending yourself.
### Text vs Binary Data
 
- **Django:** `receive(self, text_data)` only handles text frames by default. Binary data arrives via a separate `bytes_data` parameter.
- **FastAPI:** you explicitly choose what to receive ÔÇö `await websocket.receive_text()` or `await websocket.receive_bytes()`.

## 15. Sync vs Async in Django Channels -
:::info
This topic is important.
::::
 
### Why Django Examples Often Skip `async`/`await`
 
Because they use a **synchronous consumer**:
```python
class ChatConsumer(WebsocketConsumer):
    def receive(self, text_data):
        self.send(text_data=text_data)
```
No `async`, no `await` ÔÇö it runs in a **thread (blocking style)**.
 
### Django Channels Supports Two Consumer Types
 
**1. Sync Consumer**
```python
from channels.generic.websocket import WebsocketConsumer
 
class ChatConsumer(WebsocketConsumer):
    def receive(self, text_data):
        self.send(text_data=text_data)
```
 
**2. Async Consumer**
```python
from channels.generic.websocket import AsyncWebsocketConsumer
 
class ChatConsumer(AsyncWebsocketConsumer):
    async def receive(self, text_data):
        await self.send(text_data=text_data)
```
 
### Why Tutorials Often Use the Sync Version
 
- **Simplicity** ÔÇö easier to understand, no need to think about async yet.
- **Compatibility** ÔÇö Django's ORM is mostly synchronous (`User.objects.all()`), which works easily in a sync consumer.
### Why FastAPI Always Uses Async
 
FastAPI is built on ASGI with an **async-first design** ÔÇö there is no synchronous option, unlike Django.
 
```python
async def endpoint():
    await something()
```
 
### Comparison
 
| | Django Channels | FastAPI |
|---|---|---|
| Sync support | Ôťů | ÔŁî |
| Async support | Ôťů | Ôťů |
| Default style | Sync | Async |
 
### Important Rule for Async Consumers
 
If using `AsyncWebsocketConsumer`, **never** run blocking ORM calls directly:
 
```python
# ÔŁî BAD ÔÇö blocks the event loop
User.objects.all()
```
 
Instead, wrap blocking calls:
 
```python
# Ôťů GOOD
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
### When to Use Async Consumers
 
Use `AsyncWebsocketConsumer` when you have:
- Many concurrent users (e.g., chat apps)
- Real-time updates
- External API calls
- Non-blocking task requirements
**One-line clarity:** FastAPI = "everything async." Django = "you choose sync or async."

## 16. Async Database Access in Django and FastAPI

:::info
This topic is imp
::::
 
### Why DB Queries Aren't Naturally `await`-able in Django
 
Django's ORM is **synchronous (blocking)**. A call like:
```python
User.objects.all()
```
blocks the thread until the database responds. You **cannot** simply write:
```python
await User.objects.all()  # ÔŁî ERROR ÔÇö not awaitable
```
because it's not an async function and doesn't return a coroutine.
 
**What actually happens during a sync query:**
```
Python Ôćĺ DB driver Ôćĺ Database Ôćĺ wait Ôćĺ response Ôćĺ return
```
During this wait, the thread is **blocked** and can't handle other requests.
 
### Using the ORM From Async Code
 
```python
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
Internally, this moves the synchronous call to a thread pool, runs the blocking DB query there, and returns the result without blocking the main async event loop. The async app itself doesn't block ÔÇö but the DB call is still sync *internally*.

```
Your async event loop (main thread)
        Ôćô
sync_to_async grabs a separate thread from a thread pool
        Ôćô
Runs User.objects.all() on THAT thread  ÔćÉ still fully synchronous/blocking, right here
        Ôćô
That thread blocks while waiting for the DB
        Ôćô
Once it's done, the result is handed back to your async event loop
        Ôćô
Your main event loop was free to do other things the whole time

```
* Term Meaning-"Doesn't block" (the async app)-Your main event loop stays free to handle other tasks while the DB call happens elsewhere.
* Term Meaning"Sync internally" (the DB call)- The actual database query is still a normal blocking call ÔÇö it just happens on a separate worker thread instead of the main thread

* The blocking is real ÔÇö it(`sync_to_async`) didn't get magically removed. It just got moved to a thread that isn't doing anything else important, so its blockage doesn't stall the rest of your app.

### Why Django's ORM Isn't Async
 
1. **Database drivers** ÔÇö most traditional drivers (e.g., `psycopg2` for PostgreSQL) are blocking/sync by design. (A database driver is a library that lets your Python code talk to a database.)
2. **Framework design** ÔÇö Django was built before async became popular, and its entire ORM is designed around synchronous execution.
### Async Database Access in FastAPI
 
FastAPI can do fully async DB access using specialized async-native libraries:
 

| Library | What it actually is | Talks directly to the database? |
|---|---|---|
| `asyncpg` | A true DB driver ÔÇö written specifically for async PostgreSQL access | Ôťů Yes, directly |
| `asyncmy` | A true DB driver ÔÇö async MySQL driver | Ôťů Yes, directly |
| `databases` | A query library / thin wrapper ÔÇö not a driver itself | ÔŁî No ÔÇö it sits on top of drivers like `asyncpg` |
| SQLAlchemy (async mode) | A full ORM ÔÇö also not a driver itself | ÔŁî No ÔÇö it also relies on an underlying driver to actually talk to the DB |

These are:
- Ôťů Fully asynchronous
- Ôťů Work with `async`/`await`
- Ôťů Non-blocking
- Ôťů Designed for high concurrency
**SQLAlchemy** is a **hybrid ORM** ÔÇö originally synchronous, with async support added in version 1.4+.
 
```python
result = await database.fetch_all(query)
```
This is truly non-blocking, using an async DB driver.
 
### Side-by-Side
 
| | Django ORM | Async-native (e.g., SQLAlchemy async, asyncpg) |
|---|---|---|
| Async support | ÔŁî | Ôťů |
| Blocking | Ôťů | ÔŁî |
| Needs wrapper (`sync_to_async`) | Ôťů | No |
 
### Common Mistake
 
```python
class ChatConsumer(AsyncWebsocketConsumer):
    async def receive(self, text_data):
        users = User.objects.all()  # ÔŁî BAD ÔÇö blocks the event loop, kills performance
```
 
**Correct approach:**
```python
from asgiref.sync import sync_to_async
users = await sync_to_async(User.objects.all)()
```
 
### When Sync Django Is Actually Fine
 
If the app is mostly **DB-heavy with standard CRUD operations(std crud operations mostly have noraml traffic)**, a sync Django setup is often **simpler and perfectly adequate** ÔÇö async only pays off under high concurrency or many simultaneous long-lived connections.

* Async doesn't make a single DB-heavy query run faster ÔÇö the database still takes the same time to do its work. Async's benefit is letting your app stay responsive to other requests while waiting on that DB time. So "DB-heavy but normal traffic" doesn't automatically benefit from async ÔÇö only "DB-heavy and lots of simultaneous waiting requests" does.ASync helps if there simultaneous waiting requests case , not with normal traffic for "DB heavy " operations.

## 17. Authentication Fundamentals
 
**Authentication** answers: *"How does the server know who you are?"* (e.g., after you log in, the server must remember you for future requests.)
 
There are two major approaches:
 
### A. Session-Based Authentication (Traditional Django)
 
**Idea:** The server stores your login info; the browser only holds a reference to it.
 
**Flow:**
1. **Login:** username + password Ôćĺ server.
2. **Server creates a session:** `session_id = "abc123"`, stored server-side (DB/cache).
3. **Server sends a cookie:** `Set-Cookie: sessionid=abc123`.
4. **Browser sends the cookie on every request:** `Cookie: sessionid=abc123`.
5. **Server looks up the session:** `session_id Ôćĺ user`.
**Where data lives:**
 
| Location | Stores |
|---|---|
| Browser | `session_id` (cookie only) |
| Server | Full user session data |
 
**Pros:** Very secure (server controls state); easy in Django via `request.user`; built-in support.
**Cons:** Needs server memory for sessions; harder to scale across multiple servers; not ideal for APIs/mobile apps.
 
### How Django Implements This by Default
 
In `settings.py`:
```python
MIDDLEWARE = [
    ...,
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
]
```
 
Authentication is **not** done by `SessionMiddleware` alone ÔÇö it's a combination of two middlewares:
 
| Middleware | Job |
|---|---|
| `SessionMiddleware` | Reads the `sessionid` cookie, loads session data from server, attaches `request.session` |
| `AuthenticationMiddleware` | Reads `request.session`, gets the `user_id`, fetches the user, attaches `request.user` |
 
**Flow:**
```
Browser sends cookie: sessionid=abc123
        Ôćô
SessionMiddleware Ôćĺ request.session = session_data
        Ôćô
AuthenticationMiddleware Ôćĺ request.user = User object
```

* Session data is just a key-value store in server,tied to a specific user's browsing session.it is like `_auth_user_id: 5`, `_auth_user_hash: "..."`,`request.session["theme"] = "dark"` etc.

You only need to write custom code when doing custom authentication (JWT, tokens), WebSocket auth, or custom user logic.
 
**Django Channels equivalent:** WebSocket connections in Django Channels don't have a request object ÔÇö they have a scope instead. So Django's normal HTTP-only middlewares (SessionMiddleware, AuthenticationMiddleware) can't be used directly ÔÇö they're written specifically to work with request, not scope.
* **Django Channels** provides **AuthMiddlewareStack** ÔÇö think of it as: "the WebSocket-compatible version of SessionMiddleware + AuthenticationMiddleware, bundled together as one thing."

### Sessions vs Cookies ÔÇö Core Relationship
 
- A **session** is data stored on the **server** to remember a user between requests (e.g., `request.session["user_id"] = 5`), stored in a database or cache (like Redis).
- A **cookie** is a small piece of data ÔÇö typically just the `session_id` ÔÇö stored in the **browser**.
**Key rule:**
- Cookie stores only the `session_id`.
- Session (server-side) stores the actual user data.
Cookie Ôëá Session. The cookie is just a **key**; the session is the **data**.
 
**Full flow:**
1. User logs in (username + password Ôćĺ server).
2. Server creates a session: `{"abc123": {"user_id": 5}}`.
3. Server sends `Set-Cookie: sessionid=abc123` ÔÇö stored in the browser.
4. On the next request, the browser automatically sends `Cookie: sessionid=abc123` (sent as an HTTP **request header**, e.g., `Cookie: sessionid=abc123; theme=dark`).
5. Server looks up `abc123 Ôćĺ user` and sets `request.user`.
In Django, `request.session` is a Python object (interface) used to read/write session data ÔÇö
example:
```
request.session["name"] = "John"
1. You write this line in your view
        Ôćô
2. request.session (the interface object) receives this write
        Ôćô
3. Internally, it marks "name" Ôćĺ "John" as a change to be saved
        Ôćô
4. At the end of the request, Django's SessionMiddleware
   takes request.session's data and SAVES it into the
   real storage (DB row / Redis entry) tied to that session_id

```
