# Event Loop & Asyncio — Complete Notes

## 1. Event-Driven Programming

### What is it?
Instead of the traditional blocking style:
```py
do_task_1()
wait
do_taskl_2()
wait
```
Event looping programming is a program **reacts to events**- it sits idle until something happens, then responds.Events include:
- HTTP request arrives
- Timer finishes
- File becomes ready
- WebSocket message arrives
- User clicks a button


### Synchronous vs event-driven flow

**Traditional (synchronous):**
```python
task1()
task2()
task3()
```
One thing at a time.

**Event-driven:**
```python
while True:
    check_events()
    run_ready_tasks()
```
The program sleeps/waits until an event occurs, then reacts.

### The event loop
In Python's `asyncio`, the **event loop** is the heart of event-driven programming. It:
- Watches for events
- Schedules tasks
- Resumes paused coroutines

## 2. A First Asyncio Example

```python
import asyncio
 
async def hello():
    print("Hello")
    await asyncio.sleep(1)
    print("World")
 
asyncio.run(hello())
```
### What happens internally, step by step
| Step | What happens |
|---|---|
| 1 | `asyncio.run(hello())` creates the event loop |
| 2 | `hello()` creates a **coroutine object** |
| 3 | The event loop schedules it |
| 4 | Coroutine runs: `print("Hello")` |
| 5 | Hits `await asyncio.sleep(1)` → coroutine **pauses**, event loop can run other tasks |
| 6 | After 1 second: sleep event completes, coroutine resumes → `print("World")` |

---

## 3. Why Event-Driven Code Is Powerful
A **blocking threading** call like:
```python
time.sleep(5)
```
freezes the current thread for 5 seconds — nothing else in that thread can run, and the event loop can't process other tasks.

An **async** wait:
```python
await asyncio.sleep(5)
```
pauses efficiently. During that wait, other requests, WebSocket messages, and database responses can all be handled concurrently.

### Real-world example — 1000 users hitting an API
**Synchronous server:**
```
request1 -> wait DB
request2 waits
request3 waits
```
**Async event-driven server:**
```
request1 waiting DB
request2 runs
request3 runs
request4 runs
```
One thread can manage many waiting operations at once.

## 6. `gather()` Internals
`asyncio.gather()` accepts **both tasks and plain coroutines**.
```python
asyncio.gather(foo(), bar())
```
Internally, it converts the coroutines into Tasks automatically — conceptually equivalent to:
 
```python
task1 = asyncio.create_task(foo())
task2 = asyncio.create_task(bar())
await asyncio.gather(task1, task2)
```
### Direct coroutines vs explicit tasks
**Direct coroutines** — `await asyncio.gather(foo(), bar())`
✅ Cleaner, most common.
**Explicit tasks** — create tasks first, then gather them.
✅ Needed when you want **task references**, e.g. to:
- cancel later
- check status
- await later
- store the tasks

## 7. When Does a Coroutine Actually Run?
A coroutine runs only when one of these happens:
 
**1. It is directly awaited**
```python
await foo()
```
::::info
The event loop runs it directly. ❗ It is **NOT** converted into a Task.And there is no schedule. (Common misconception — this is important.)
* There is no task creation, zero scheduling, zero queue placement, and zero loop interaction when `await child()` is called. It is a direct, immediate, inline execution.
* **The Event Loop  manages independent Tasks**. It does not manage the internal sub-coroutine calls inside an active Task.
*  Explanation: when you do `await foo()`, the event loop doesn't create a separate Task object to track foo() — instead, the current Task just keeps running foo()'s code as part of itself. No new scheduling entry is created.And waiting and resuming is considering as in the same Task.
::::

**2. It's wrapped in a Task**
```python
asyncio.create_task(foo())
```
Now the coroutine becomes a Task.

**3. It's passed to `gather()`**
```python
await asyncio.gather(foo(), bar())
```
`gather()` converts coroutines → Tasks automatically.

**4. It's passed to `asyncio.run()`**

 
```python
asyncio.run(work())
```
Internally, Python roughly does:
```python
loop = asyncio.new_event_loop()      # creating event loop
coroutine = work()                   # creating coroutine object
task = loop.create_task(coroutine)   # wraps coroutine into a Task
# Now the event loop can manage it, because it only schedules Tasks/Futures
loop.run_until_complete(task)        # loop starts running
```
 
> **Key point:** `await foo()` does **not** create a Task — this is the central misunderstanding many people have.

## 8. Event Loop Internals — The Big Picture
When you write:
```python
asyncio.run(main())
```
Python:
1. Creates an event loop
2. Starts running `main()`
3. Keeps switching between coroutines whenever they `await`
4. Stops when everything finishes
**Normal synchronous code:** `a(); b(); c()` — the CPU waits for each function to fully complete.
 
**Async code:** `await a(); await b(); await c()` — if `a()` is waiting on I/O (network, database, sleep, websocket, file), the event loop pauses it and runs something else. That's the entire async model.

| Component | Purpose |
|---|---|
| Ready Queue | Tasks ready to execute now |
| Waiting Tasks | Tasks waiting for I/O/timers |
| Selector/Poller | OS mechanism to detect events |
| Task Objects | Wrap coroutines |
| Callbacks | Functions stored now, run later when an event happens |

 
**Example — timer callback:**
```python
loop.call_later(5, hello)
```
Meaning: "After 5 seconds, run `hello()`." The callback is stored now but executed later.

### Coroutine vs Task — calling a coroutine doesn't run it

```python
async def foo():
    await asyncio.sleep(1)
 
foo()   # does NOT run it — only creates a coroutine object
```
 
- `await foo()` → the event loop runs it **directly**, with no scheduling like Tasks/Futures. The current coroutine pauses and directly runs `foo()` until `foo()` itself hits an `await`.
- `task = asyncio.create_task(foo())` → now the event loop manages it as a Task that **wraps** the coroutine. The loop only schedules **Tasks/Futures**, never raw coroutine functions.

## 9. Step-by-Step Internal Flow
 
```python
import asyncio
 
async def work():
    print("Start")
    await asyncio.sleep(2)
    print("End")
 
asyncio.run(work())
```
1. **Create event loop** — `loop = EventLoop()`
2. **Convert coroutine into a Task** — `task = Task(work())`
3. **Put task into the ready queue** — `ready_queue = [task]`


### The event loop's main cycle
 
```python
while not stopped:
    run_ready_tasks()              # from ready queue
    wait_for_io_events()
    move_completed_io_tasks_to_ready()
```
This runs thousands of times per second.

### Running a task
 
The event loop picks a task from the queue and calls `task._step()`, which does **both**:
1. Starts coroutine execution the first time, **or**
2. Resumes coroutine execution later (after an `await`)
Your coroutine runs until it **finishes** or hits an `await`.

### What happens at `await`
 
```python
await asyncio.sleep(2)
```
The coroutine effectively says: *"I cannot continue now. Resume me later."* The task pauses:
```
Task state: RUNNING -> WAITING
```
The event loop removes it from the ready queue.

### `sleep()` internals
 
`asyncio.sleep(2)` creates a timer. The loop stores `resume this task after 2 seconds` in an internal **timer heap**.

### When the loop goes idle
 
If the ready queue is empty, the event loop calls the OS selector:
 
| OS | Mechanism |
|---|---|
| Linux | epoll |
| Mac | kqueue |
| Windows | IOCP |
 
The loop effectively asks the OS: *"Wake me when a socket is ready, a timer expires, or data arrives."*

### When the timer completes
 
After 2 seconds, the OS/timer wakes the event loop, which moves the task back into the ready queue. The loop calls `task._step()` again, the coroutine continues after the `await` (`print("End")`), and then finishes.

### Visual timeline
 
```
Time →
────────────────────────────────
Task starts
  | print("Start")
  | await sleep(2)
  V
Task paused
Event loop runs other tasks or sleeps
2 sec later
  V
Task resumed
  | print("End")
Task finished
```

## 10. CPU Usage: Why Async Is Efficient (Busy-Waiting vs Sleeping)
 
**Busy-waiting / spinning** means the CPU continuously executes checking code, even when nothing has happened:
```python
while True:
    check_if_done()
```
This can run millions of times per second, repeatedly checking a variable — extremely inefficient. The CPU never truly idles.
 
**Asyncio does NOT do this.** When nothing is ready (no socket data, no expired timers, no ready tasks), the event loop tells the OS: *"Put me to sleep until something happens,"* roughly via:
```python
selector.select(timeout=5)
```
This blocks **efficiently** — the Python thread sleeps inside an OS syscall, and the CPU is free for other work. No repeated Python-level checking happens.
 
So: **event loop sleeping ≠ CPU fully busy.** CPU usage stays very low while waiting.
### How does it wake up?
 
An OS/hardware interrupt wakes the sleeping selector. Then `selector.select()` returns, the event loop runs again, and the task resumes.
 
> **Correct statement:** the OS selector/wait system, triggered by a hardware/OS event, wakes the sleeping event loop thread.

## 11. Real I/O Example
 
```python
response = await session.get(url)
```
 
Internally:
1. The socket is registered with the OS selector
2. The task pauses
3. The loop runs other tasks
4. The OS reports the socket has data
5. The loop resumes the task
This is the heart of async networking — the event loop hands the "waiting" job to the OS selector; when data arrives, the OS informs the selector, and meanwhile the event loop may be running other tasks (or sleeping, if the ready queue is empty).

### Key internal structures
 
- **Ready Queue** — `deque([task1, task2])`, tasks ready immediately
- **Scheduled Timers** — usually a min-heap ordered by time: `[(resume_at, task)]`
- **Selector** — tracks sockets/file descriptors: `socket -> task waiting`

## 12. The Two Waiting Types: I/O vs Timers
 
Asyncio's event loop waits for two kinds of events:
 
| Type | Example |
|---|---|
| I/O events | socket data arrived |
| Timer events | sleep timeout expired |

### I/O waiting
 
```python
await socket.recv()
```
The task waits for socket readiness; the OS selector tracks the file descriptor. Wakeup chain: hardware/network interrupt → OS marks socket ready → selector wakes.

**NOte**- A file descriptor (FD) is just a number (an integer) that the operating system uses as a handle to refer to an open "thing" — could be a file, a socket, a pipe, a device, etc.

### Timer waiting
 
```python
await asyncio.sleep(5)
```
This is **not** waiting for network I/O. Instead, asyncio stores `(resume_at_time, task)` — e.g. `(10:30:05, task1)` — usually inside a **min-heap**, so the smallest wakeup time stays on top. ("Timer heap" and "min-heap" refer to the same structure here.) This tells the event loop: *"When is the next timer due?"

### Both waits happen together at syscall(selctor)
 
If the next timer in the timer heap expires in 5 seconds, the event loop does:
```python
selector.select(timeout=5)
```
meaning: *"Sleep until an I/O event occurs OR the 5-second timeout expires"* — whichever comes first. The single OS waiting syscall (linux -> .select()) handles **both** I/O waiting and timer-timeout waiting together.

**Example**:
```py
async def task_sleep():
    await asyncio.sleep(5)     # Task X — pure timer, no I/O

async def task_network():
    await socket.recv()       # Task Y — pure I/O, no timer
```
* Task X (asyncio.sleep(5)) — this task itself is **not** registered with the OS selector at all. It doesn't touch any file descriptor. It only adds an entry to asyncio's own internal timer heap: `(wake_at=T+5, Task X)`. The selector doesn't know Task X exists.
* Task Y (socket.recv()) — this task is registered with the OS selector. Its socket's file descriptor gets registered: "tell me when FD 7 is readable." The timer heap doesn't know Task Y exists.
* Now — the event loop has to wait for either of these to become ready. It only has one mechanism for blocking efficiently: the OS selector call. So it does this:
  * Looks at the timer heap → finds the soonest timer → Task X needs 5 seconds.
  * Calls selector.select(timeout=5) — registered FDs include Task Y's socket (FD 7). The timeout=5 value comes from Task X's timer, purely so the call doesn't block forever if Task Y's socket never becomes ready.
* So this one syscall is simultaneously:

  * watching Task Y's socket (real I/O registration)
  * bounded by Task X's timer (just a numeric timeout, no FD involved)
* If Task Y's socket gets data after 2 seconds → select() returns early at 2s, with Task Y's FD in the ready list. Task X's 5-second timer hasn't expired — Task X stays in the timer heap, untouched.
* If nothing arrives on Task Y's socket for the full 5 seconds → select() returns at 5s with an empty I/O list. The event loop checks the timer heap, sees Task X's timer expired, moves Task X to ready.Task Y is not expired, not cancelled, not finished, not touched at all. It's still just sitting there, waiting, exactly as before.
* **Summary**
```
* asyncio.sleep(5) is not waiting for I/O — true, for that specific task. It never touches a file descriptor or the selector's registration list.


* selector.select() handles I/O and timeouts together — also true, but at the event loop level, across all tasks combined. The selector call's timeout parameter is just borrowed from the timer heap's soonest entry — it doesn't mean the sleep itself becomes an I/O operation.
```
### Resuming after the timeout
 
If no socket event occurs, after 5 seconds the OS timeout expires, `selector.select()` returns, the event loop checks the timer heap ("timer expired"), moves the task to the ready queue, and resumes the coroutine.
 
> **Bottom line:** the timer heap is just asyncio's internal structure for knowing how long `selector.select(timeout=...)` should sleep. The actual wakeup always comes through the OS waiting system — triggered either by I/O readiness or by timeout expiration.

* `selector.select()` always returns a list of ready events - here if no socket event happens , it will return [].

## 13. Task State Machine
 
```
CREATED
  ↓
READY
  ↓
RUNNING
  ↓
WAITING (await)
  ↓
READY
  ↓
FINISHED
```
### Why `await` gives control back
 
`await` means: *"Suspend THIS coroutine and let the event loop run others."*
 
Without any `await` (e.g. `while True: pass`), the event loop **freezes**, because the coroutine never yields control.

### Cooperative multitasking
 
Asyncio uses **cooperative multitasking** — tasks voluntarily pause using `await`. Unlike threads:
- No OS thread switching
- No true parallel execution (usually)
- One thread runs many coroutines
  
### Single-threaded by design
 
```
1 thread
1 event loop
many coroutines
```
Only one coroutine executes at any instant. Switching only happens at `await` points.

### Simplified internal pseudocode
 
```python
while True:
    # run ready tasks
    for task in ready_queue:
        task.run_until_next_await()
    # check timers
    move_expired_timers_to_ready()
    # wait for socket events
    events = selector.select()
    # wake waiting tasks
    for event in events:
        ready_queue.append(event.task)
```
### Why `create_task()` matters
 
`await foo()` runs **sequentially**. But `asyncio.create_task(foo())` tells the loop to *schedule it independently in the background*, so multiple tasks can progress together.

## 14. Futures and Tasks

### What is a Future?
 
A **Future** is a low-level placeholder object representing *"a value/result that will be available later."* Think of it as an empty box now — the value comes later.
 
A Future internally stores:
- pending/done state
- result
- exception
- callbacks

```python
future = asyncio.Future()
# Future: PENDING
 
future.set_result(100)
# Future: DONE, Result: 100
```

### Why Futures exist
 
Suppose a coroutine does:
```python
await socket.recv()
```
The socket data isn't ready immediately, so asyncio creates a Future representing *"future socket data."* The task pauses, waiting for that Future. When the data arrives:
```
OS wakes selector → asyncio gets data → future.set_result(data)
```
Now the waiting task resumes.
 
Not everything is a coroutine — sockets, callbacks, and external events also need a generic way to represent "something finishes later." Futures unify all of these.

### Awaiting a Future directly
 
```python
result = await future
```
If the result isn't ready, the current task pauses and the event loop runs others. When the future gets its result, the task resumes and `await` returns the value.
 
Who actually uses Futures directly? Mostly low-level asyncio internals: socket operations, transport/protocol APIs, event loop callbacks, and bridging callback-style code into async code.

 
### Task is a special Future
 
```
Task IS-A Future
isinstance(task, asyncio.Future)  # True
```
 
```
Future
  ↑
 Task
```
Task **inherits** Future's behavior (result, done state, exception, callbacks) **plus** its own(task's) coroutine execution and scheduling.
 
| | Future | Task |
|---|---|---|
| What it is | Just a container for a future result | Actively executes a coroutine inside the event loop |
| Runs code? | No | Yes |

### Examples
 
**Future, completed manually:**
```python
import asyncio
 
async def main():
    future = asyncio.Future()
    future.set_result("hello")
    result = await future
    print(result)
 
asyncio.run(main())
# Output: hello
```
The Future itself didn't execute anything — you completed it manually.
 
**Task, runs a coroutine automatically:**
```python
import asyncio
 
async def work():
    await asyncio.sleep(1)
    return 42
 
async def main():
    task = asyncio.create_task(work())
    result = await task
    print(result)
 
asyncio.run(main())
```
The Task automatically runs the coroutine and eventually stores the result internally, just like a Future.

### How a Task works internally
 
```python
task = asyncio.create_task(work())
```
Roughly:
```
Task contains: coroutine object, state, result, callbacks
```
The event loop repeatedly calls `task._step()` until the coroutine finishes. Then the task internally does `future.set_result(return_value)` — which is exactly why a Task *behaves* like a Future.
 
Because Task inherits Future, `await task` means *"wait until this task's underlying future becomes DONE."*

 
| | Defines/does |
|---|---|
| **Coroutine** | Defines async work |
| **Future** | Placeholder for a result |
| **Task** | Runs the coroutine **and** stores the result |
 
One-line definitions:
- **Future** — "I will eventually contain a result."
- **Task** — "I am executing a coroutine and will eventually contain its result."


### `gather()` and Futures
 
```python
await asyncio.gather(a(), b())
```
Internally: coroutines → Tasks, and those Tasks are tracked via Futures.

## 16. uvloop
 
### What is it?
 
`uvloop` is a **faster replacement** for Python's default asyncio event loop. It's built on top of **libuv** — the same high-performance library used internally by Node.js.
 
### Default asyncio vs uvloop
 
```python
# Default
import asyncio
asyncio.run(main())   # uses asyncio's built-in DefaultEventLoop
```
 
```python
# With uvloop
import asyncio
import uvloop
asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())
asyncio.run(main())
```
Your async code stays mostly the same — only the engine underneath changes.
 
Shortcut form:
```python
import asyncio
import uvloop
uvloop.install()
asyncio.run(main())
```
(`uvloop.install()` is shorthand for setting the event loop policy.)
 
### Why it's faster
 
The default asyncio loop is implemented mostly in Python. `uvloop` is implemented in **C**, using **libuv** and optimized system calls — so socket handling, scheduling, callbacks, timers, and polling all become much faster.
 
**Architecture comparison:**
```
Normal asyncio:        uvloop:
Python asyncio loop    Python asyncio APIs
  ↓                      ↓
selectors              uvloop
(epoll/kqueue)           ↓
  ↓                    libuv (C library)
OS                       ↓
                       OS
```
 
libuv handles epoll (Linux), kqueue (Mac), IOCP (Windows), timers, async sockets, a thread pool, and file I/O very efficiently in C.
 
**Result:** lower latency, higher throughput, less overhead — especially for web servers, websockets, APIs, and high-concurrency networking. Frameworks like FastAPI, aiohttp, and Uvicorn commonly recommend it.
 
```bash
pip install uvloop
```

### Important notes
 
uvloop is **not** a new async syntax, a different coroutine system, or a framework — it only replaces the event loop engine underneath asyncio.
 
**Analogy:** your async code is the driver, the event loop is the car engine. uvloop gives you a faster engine — same driver, same road, better performance.
 
**Limitation:** uvloop mainly improves I/O-bound async applications. It does **not** speed up CPU-heavy calculations, blocking synchronous code, or bad async patterns.
 
| | Default asyncio | uvloop |
|---|---|---|
| Official Python | Yes | No |
| Built into Python | Yes | No |
| Modern | Yes | Yes |
| Speed | Good | Usually faster |
| Uses C/libuv | No | Yes |
| Best for beginners | Yes | Not needed initially |
| Production performance | Good | Excellent |

## 17. Important Asyncio Patterns
 
These patterns are common ways to structure async programs to manage concurrency, scheduling, cancellation, coordination, background jobs, and producer/consumer systems.
 
### 1. Sequential await pattern
 
```python
async def main():
    await task1()
    await task2()
    await task3()
```
Execution: task1 finishes, then task2, then task3 — **no concurrency**. Use this when each task depends on the previous result.

### 2. Concurrent pattern — `create_task()`
 
```python
async def main():
    t1 = asyncio.create_task(task1())
    t2 = asyncio.create_task(task2())
    await t1
    await t2
```
Both tasks run **at the same time**.
 
`await t1` means *"pause until task t1 finishes"* — **not** *"start the task."* The task already started earlier, the moment `create_task()` ran:
1. Coroutine object created
2. Task object created
3. Task added to the event loop's ready queue (`ready_queue.append(task)`)
Whenever the event loop next gets control (at any `await` point), it begins executing it.

### 3. `gather()` pattern
 
The most common concurrency pattern:
```python
results = await asyncio.gather(
    task1(),
    task2(),
    task3()
)
```
Runs all concurrently and returns results **in order**. If you pass coroutines, they're automatically converted into Tasks — you can pass tasks, coroutines, or futures into `gather()`.
 
Equivalent (but more verbose) to show the order:
```python
t1 = create_task(task1())
t2 = create_task(task2())
await t1
await t2
```
Use this when you need all the results together.

### 4. Fire-and-forget background task
 
```python
asyncio.create_task(background_worker())
```
No `await` — the task runs independently.
 
```python
async def logger():
    while True:
        await asyncio.sleep(5)
        print("logging")
 
asyncio.create_task(logger())
```
 
⚠️ **Danger:** `asyncio.run()` keeps the event loop alive only until the **main coroutine** completes — not until every background task finishes. When the loop shuts down, remaining tasks get **cancelled**. So `asyncio.create_task(...)` does **not** guarantee task completion — if nobody awaits it and the program exits, the task may die unfinished.

### 5. Producer–consumer pattern
 
Uses `asyncio.Queue()`:
```python
import asyncio
 
queue = asyncio.Queue()
 
async def producer():
    for i in range(5):
        await queue.put(i)
 
async def consumer():
    while True:
        item = await queue.get()
        print(item)
        queue.task_done()
```
Flow: **Producer → Queue → Consumer**
 
A queue is a temporary storage structure for passing data between producers and consumers, following **FIFO order** (First In → First Out). Real uses: message systems, websocket processing, job workers, API pipelines.
 
**Walking through `get()`/`put()`:**
```python
await queue.put(10)
await queue.put(20)
# Queue: [10, 20]
 
item = await queue.get()   # -> 10
# Queue: [20]
 
# next get() -> 20
```
 
**Why `await`?** Because the queue may be **empty** when `get()` is called — the consumer pauses until the producer adds something. Likewise, `queue.put()` is awaitable because the queue might be **full**.
 
By default, `asyncio.Queue()` is unbounded, so `await queue.put(x)` returns immediately. But you can bound it:
```python
queue = asyncio.Queue(maxsize=2)
await queue.put(1)
await queue.put(2)   # Queue: [1, 2] — now full
 
await queue.put(3)   # cannot insert immediately — producer pauses
```
The producer task becomes `WAITING` until a consumer removes an item via `await queue.get()`, freeing up space; then the producer wakes and continues.
### 6. Worker pool pattern
 
Multiple workers consume the **same** queue:
```python
workers = [
    asyncio.create_task(worker())
    for _ in range(5)
]
```
Now 5 workers process jobs concurrently. Used in crawlers, API fetchers, and task-processing systems.
 
> A **crawler** is a program that automatically visits many web pages and collects information — e.g. starting at `https://example.com`, it downloads the page, extracts links, visits those links, and repeats.

### 7. Semaphore pattern (limit concurrency)
 
Without limits, `await asyncio.gather(*many_requests)` could fire off 10,000 requests at once — a bad idea. Use a semaphore instead:
```python
sem = asyncio.Semaphore(10)
 
async def fetch():
    async with sem:
        await api_call()
```
Only 10 concurrent tasks are allowed at a time. Used for rate limits, DB connections, and API throttling.

### 8. Timeout pattern
 
Prevent infinite waiting:
```python
await asyncio.wait_for(task(), timeout=5)
```
If the task exceeds the timeout, it's cancelled and a `TimeoutError` is raised.

### 9. Cancellation pattern
 
```python
task.cancel()
```
Inside the coroutine:
```python
try:
    await something()
except asyncio.CancelledError:
    cleanup()
    raise
```
Cancellation is **cooperative** — the task stops at the next `await`.

### 10. Event pattern
 
`asyncio.Event` is used for **signaling** between coroutines. One coroutine says *"wait until something happens,"* another says *"okay, it happened now."*
 
```python
event = asyncio.Event()        # initially NOT SET
 
await event.wait()             # pauses if not set
event.set()                    # wakes all waiters
event.clear()                  # resets to NOT SET
```
 
Use cases: coordination, readiness notification, startup synchronization.
 
```python
import asyncio
 
event = asyncio.Event()
 
async def waiter():
    print("Waiting for signal...")
    await event.wait()
    print("Signal received!")
 
async def sender():
    await asyncio.sleep(2)
    print("Sending signal")
    event.set()
 
async def main():
    await asyncio.gather(waiter(), sender())
 
asyncio.run(main())
```
Output:
```
Waiting for signal...
(after 2 sec)
Sending signal
Signal received!
```
 
**Internal flow:** `event` starts `False`. `waiter()` calls `await event.wait()`; since it's false, the waiter task pauses. After 2 seconds, `sender()` calls `event.set()`, making it `True` — the event loop wakes **all** tasks waiting on that event. Note that `event.set()` does **not block**; it simply signals "you may continue now." Multiple coroutines can wait on the same event, and all of them wake up together when it's set.

### 11. Lock pattern
 
Prevent race conditions:
```python
lock = asyncio.Lock()
 
async with lock:
    shared_data += 1
```
Only one coroutine enters the critical section at a time. This is needed because coroutines can switch at any `await` point.
 
**What is a race condition?** It happens when multiple tasks access/modify shared data at the same time, and the final result depends on execution order — causing wrong values, lost updates, or inconsistent state.
 
**Example:** suppose `counter = 0` and two coroutines both do `counter += 1`. You'd expect `2`, but you might get `1`. That's a race condition.
 
**Why it happens:** `counter += 1` is **not** one atomic operation — internally it's roughly:
1. Read counter
2. Calculate new value
3. Write counter
If a task switch (due to an `await`) happens in the middle, before the write, bad things happen.

### 12. TaskGroup pattern (modern Python, 3.11+)
 
Modern **structured concurrency**:
```python
async with asyncio.TaskGroup() as tg:
    tg.create_task(task1())
    tg.create_task(task2())
```
Safer than raw `create_task`. **Benefit:** if one task fails, all related tasks are cancelled automatically, with better cleanup. Python 3.11+ prefers `TaskGroup` over unmanaged background tasks.
 
### 13. Pipeline pattern
 
Chain async stages: **Fetch → Parse → Save**. Each stage is async, often connected via queues.

### 14. Fan-out / fan-in pattern
 
These names come from flow-diagram visuals — how work spreads out and then comes back together.
 
**Fan-out** — one thing splits into many tasks, like a fan opening outward:
```
       Task1
      /
Main ──┼── Task2
      \
       Task3
```
```python
await asyncio.gather(
    fetch(url1),
    fetch(url2),
    fetch(url3)
)
```
One main flow spreads into many concurrent tasks — that's fan-out.
 
**Fan-in** — after many tasks finish, results come back into one place:
```
Task1 ─\
Task2 ──┼── Final Result
Task3 ─/
```
```python
results = await asyncio.gather(
    fetch(url1),
    fetch(url2),
    fetch(url3)
)
```
All results combine into `results` — that combining-back step is fan-in. Used in parallel API calls, distributed requests, and crawlers.

### 15. Retry pattern
 
```python
for _ in range(3):
    try:
        return await api_call()
    except Exception:
        await asyncio.sleep(1)
```
Common in networking. Walking through it:
 
- **Attempt 1:** `api_call()` fails (e.g. `ConnectionError`, `TimeoutError`) → jumps to `except Exception:` → `await asyncio.sleep(1)` waits 1 second → loop continues.
- **Attempt 2:** tries again. If it fails again, same except/sleep cycle.
- **Attempt 3:** same process.
- **On any success:** `return await api_call()` returns the value immediately — the loop exits, no more retries.
Visual:
```
Attempt 1 → failed
  ↓ wait 1 sec
Attempt 2 → failed
  ↓ wait 1 sec
Attempt 3 → success
  ↓
return result
```
 
### 16. Backpressure pattern
 
Prevent overload using a bounded queue:
```python
queue = asyncio.Queue(maxsize=100)
```
The producer pauses if the queue is full. Very important in real systems.

### 17. Shield pattern
 
Protect a task from outer cancellation:
```python
await asyncio.shield(task)
```
An outer cancellation won't stop (cancel) the protected task.
 
### Putting it together — a common real architecture
 
```
Producer
  ↓
Queue
  ↓
Worker Pool
  ↓
Semaphore-limited HTTP requests
  ↓
Gather results
```
 
### Patterns to master first
 
1. `await`
2. `create_task`
3. `gather`
4. Queue producer-consumer
5. Semaphore
6. Cancellation
7. TaskGroup

## 18. Worked Example: Async Connection Pool
 
**Problem without a connection pool:** if 100 users hit your site and every request creates a new DB connection —
```python
connect to postgres
run query
disconnect
```
— this is expensive, because creating a DB connection involves a TCP handshake, authentication, memory allocation, and DB process/thread setup. Doing this thousands of times repeatedly is slow.
 
**Solution:** create a handful of connections once and reuse them:
```python
conn = await create_db_connection()
# Pool: conn1, conn2, conn3
```
Tasks borrow one (`conn = await pool.get()`) and return it when done (`await pool.put(conn)`).
 
Purpose: avoid expensive connection creation, limit max connections, improve performance. Real libraries already do this internally — Django's DB connections, SQLAlchemy's pool, asyncpg's pool, aiomysql's pool.
 
### Full example
 
```python
import asyncio
 
class Connection:
    def __init__(self, id):
        self.id = id
 
    async def query(self):
        await asyncio.sleep(1)
        print(f"Query using connection {self.id}")
 
class AsyncConnectionPool:
    def __init__(self, size):
        self.pool = asyncio.Queue()
        for i in range(size):
            conn = Connection(i)
            self.pool.put_nowait(conn)
 
    async def acquire(self):
        print("Waiting for connection...")
        conn = await self.pool.get()
        print(f"Acquired connection {conn.id}")
        return conn
 
    async def release(self, conn):
        print(f"Released connection {conn.id}")
        await self.pool.put(conn)
 
async def worker(name, pool):
    conn = await pool.acquire()
    try:
        print(f"{name} started work")
        await conn.query()
        print(f"{name} finished work")
    finally:
        await pool.release(conn)
 
async def main():
    pool = AsyncConnectionPool(size=2)
    await asyncio.gather(
        worker("A", pool),
        worker("B", pool),
        worker("C", pool),
        worker("D", pool),
    )
 
asyncio.run(main())
```
* Here 4 workers using 2 connections from connection pool
### What happens here? (pool size = 2)
 
- **Initially:** Queue = `[conn0, conn1]`
- **A starts** → `conn = await pool.acquire()` → gets `conn0`
- **B starts** → gets `conn1`
- **C starts** → queue empty → `await self.pool.get()` **suspends** C; it waits asynchronously
- **A finishes** → `await pool.release(conn0)` puts the connection back → C wakes up and gets it
### Interview-style concepts
 
**Why a Queue for the pool?**
- Thread-safe/async-safe coordination
- Automatic waiting
- FIFO ordering
**Why `await pool.get()`?** If no connection exists, the coroutine pauses, the event loop runs other tasks, and there's no CPU blocking.
 
**Why `finally`?** Very important — even if `query()` crashes:
```python
finally:
    await pool.release(conn)
```
ensures the connection returns to the pool. Otherwise, a **connection leak** can happen.
 
**Note on `put_nowait()`:** `self.pool.put_nowait(conn)` means *"put this item into the queue immediately without waiting."* In a connection pool: the **connections/resources** are the pool, and the **queue** stores the available connections.
 
```
Pool of connections:
--------------
conn1
conn2
conn3
--------------
```
Usually implemented internally using `asyncio.Queue()`.
 
So here: **Pool** = the reusable connections (`conn1, conn2, conn3` — the actual resources). **Queue** = the data structure managing which connections are currently available, e.g. `self.pool = asyncio.Queue()` containing `[conn1, conn2, conn3]`.

## 19. Connection Pool vs Worker Pool
 
A **worker pool** manages **workers doing jobs/tasks** — e.g. 3 workers processing 100 jobs. Workers continuously pull from a queue (`job = await queue.get()`) and process it. Purpose: limit concurrency, distribute workload.
 
Structurally, a connection pool looks very similar to a worker pool, because both use a shared queue/resource, waiting tasks, and limited concurrency. But the **meaning** is different.
 
In a **worker pool**, workers are permanent/background consumers:
```python
jobs = asyncio.Queue()
 
# each worker:
while True:
    job = await jobs.get()
    await process(job)
```
Flow: **Job Queue → Workers process jobs.** The workers themselves are the pool. So: the **workers** are the pool, and the **jobs/tasks** are stored in the queue.
 
```
 Job Queue
 -------------
 job1
 job2
 job3
 -------------
   ↓     ↓
 Worker1 Worker2 Worker3
```
 
### Full worker pool example
 
```python
import asyncio
 
queue = asyncio.Queue()
 
async def worker(name):
    while True:
        job = await queue.get()
        print(f"{name} processing {job}")
        await asyncio.sleep(2)
        print(f"{name} finished {job}")
        queue.task_done()
 
async def main():
    workers = [
        asyncio.create_task(worker("W1")),
        asyncio.create_task(worker("W2")),
    ]
    for i in range(5):
        await queue.put(f"job-{i}")
    await queue.join()
    for w in workers:
        w.cancel()
 
asyncio.run(main())
```
* Here 2 workers processing 5 jobs in the worker pool.

### Visual difference
 
**Connection Pool:**
```
Task A → conn1
Task B → conn2
Task C waits
```
 
**Worker Pool:**
```
Worker1 ← job1
Worker2 ← job2
job3 waits
```
 
### Easy way to remember
 
| | Queue = | Pool = |
|---|---|---|
| **Worker Pool** | work waiting | workers |
| **Connection Pool** | available connections | connections themselves |
 
### Side-by-side comparison
 
| | Pending/Available | The "pool" itself |
|---|---|---|
| **Worker Pool** | Pending jobs/tasks | Workers |
| **Connection Pool** | Available connections | Reusable connections/resources |
 
| | Worker Pool | Connection Pool |
|---|---|---|
| What's pooled | Task executors | Reusable resources |
| Example | Background workers | DB connections |
| Relationship | Workers consume jobs | Tasks borrow resources |
| Concern | Work management | Resource management |
 
Both commonly use `asyncio.Queue`, semaphores, and the producer-consumer pattern — which is why they feel conceptually similar.

## 20. Queue vs Pool — They Are NOT the Same Thing
 
> A queue is **not always** a pool. They are different concepts.
 
### Queue
 
A queue is just a **data structure**. It stores things in order. It can hold jobs, messages, connections, events — anything. Purpose: temporary waiting/storage.
 
```python
queue = asyncio.Queue()
```

### Pool
 
A pool is a **group of reusable, limited resources** — e.g. a worker pool, connection pool, or thread pool. Purpose: reuse limited resources efficiently.

### Pools often use queues internally
 
It's very common for a pool to be implemented internally using a queue, because a queue helps manage waiting, coordination, and availability. **But the queue itself is not the pool.**

### Example 1 — a plain queue (NOT a pool)
 
```python
queue = asyncio.Queue()
await queue.put("job1")
await queue.put("job2")
```
This is just a queue of jobs. No pool exists here.
 
### Example 2 — a connection pool
 
```python
pool = asyncio.Queue()
pool.put_nowait(conn1)
pool.put_nowait(conn2)
```
Here, the **connections** are the pool; the **queue** manages which connections are currently available.

### Key difference: `put_nowait()` vs `await put()`
 
| | `put_nowait()` | `await put()` |
|---|---|---|
| Timing | Immediate | Async operation |
| Can suspend? | No | Yes, if the queue is full |
| Typical use | During setup, when you **know** the queue isn't full | Inside async runtime flow |

> **Suspend** means the function pauses execution and gives control back to the event loop, without blocking the whole program.

### Example 3 — a worker pool
 
```python
workers = [
    create_task(worker()),
    create_task(worker()),
]
```
Here, the **workers** are the pool; the **queue** (the queue is used inside the worker's logic (`await queue.get()` appears inside worker()), not that the queue object is created/scoped inside the function( worker()).) stores the jobs.

### Easy rule of thumb
 
- **Queue answers:** *"What is waiting?"*
- **Pool answers:** *"What reusable, limited resources exist?"*
