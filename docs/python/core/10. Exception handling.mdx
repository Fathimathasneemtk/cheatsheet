*   **Exception handling** allows a program to handle unexpected errors during **execution** in a controlled way,instead of crashing it abruptly.
* eg: Handles runtime errors such as invalid input, file not found, type mismatches,division by zero that occur during runtime.(run and execution are same meaning in many languages)
* Compile time errors found by compiler/interpreter before program starts running eg : (Syntax,Missing parenthesis).So **Exception handling** deals with only **runtime** erros.

## Exception chaining- Implicit chaining `__context__` and explicit chaining  `__cause__`
### 1. Implicit chaining : `__context__`
* When a new exception is raised while python handling another exception,Python automatically links the new exception to the active exception through `__context__`. 
```py
try:
    int("hello")
except ValueError:
    raise TypeError("Conversion failed")

```
Python is automatically doing something follows
```py
type_error.__context__ = value_error 

```

The traceback displays
```
ValueError: invalid literal for int() with base 10: 'abc'

During handling of the above exception, another exception occurred:

TypeError: Conversion failed

```
So Implicit chainig does show a relationship: The TypeError occured while handling the `ValueError`.

### 2.Explicit chaining : `__cause__` 

Using `raise ... from e` says that the new exception happened specifically **because of e**.
```py
try:
    int("abc")
except ValueError as e:
    raise TypeError("Conversion failed") from e

```
Python sets:
`
new_exception.__cause__=e
`

Traceback displays:

```
ValueError: invalid literal for int() with base 10: 'abc'

The above exception was the direct cause of the following exception:

TypeError: Conversion failed
```

```py
try:
    int("abc")
except ValueError as e:
    try:
        raise TypeError("Conversion failed") from e
    except TypeError as ex:
        print(ex.__cause__ is e)          # True
        print(ex.__context__ is e)        # True

```

::::info
* __context__: happened **while handling** the previous exception.
* __cause__: happened **because of** the previous exception.
* The distiction is 
  ```
    Implicit chaining:
    new_exception.__context__ = active_exception

    Explicit chaining:
    new_exception.__cause__ = specified_exception
  ```
* Traceback messages are 
```py
    try:
        int("abc")
    except ValueError as e:
        try:
            raise TypeError("Conversion failed") from e
        except TypeError as ex:
            print(ex.__cause__ is e)          # True
            print(ex.__context__ is e)        # True
```

## Best Practices
### 1.  Be Specific — Catch Exact Exceptions
Never use a bare except: or overly broad except Exception (You dont know which one to catch). Catch only what you expect and can handle.

```py
# ❌ Bad
try:
    data = int(user_input)
except:
    pass

# ✅ Good
try:
    data = int(user_input)
except ValueError:
    print("Input must be a number")

```
### 2.Dont swallow exceptions silently.
A bare pass hides bugs. At minimum, log the error.

```py
# ❌ Bad
except Exception:
    pass

# ✅ Good
import logging
except Exception as e:
    logging.exception("Unexpected error: %s", e)
    raise  # re-raise if you can't recover
```

### 3. Usually finally for cleanup
finally runs whether or not an exception occurred — ideal for releasing resources.

```py
f = open("data.txt")
try:
    process(f)
except IOError as e:
    logging.error("File error: %s", e)
finally:
    f.close()  # always runs

```
Even better, use context managers (with) which handle this automatically:
```py
with open("data.txt") as f:
     process(f)
```

### 4. Create Custom Exception Classes
For libraries or large apps, define your own exception hierarchy so callers can catch precisely.

```py
class AppError(Exception):
    """Base exception for this app."""

class DatabaseError(AppError):
    """Raised on DB failures."""

class ValidationError(AppError):
    """Raised on invalid input."""

```
Here for exception we are giving semantic meaning

### 5.  Raise with Context — Use `raise ... from`
Preserve the original traceback(Python's way of showing you exactly where and how an error occurred) when wrapping exceptions.

```py
try:
    connect_to_db()
except ConnectionRefusedError as e:
    raise DatabaseError("Could not reach database") from e

```
### 6. Avoid Using Exceptions for Flow Control(if else)
Exceptions are for exceptional situations, not regular branching logic(if else ).

```py
# ❌ Bad (using exceptions as if/else cases)
try:
    value = my_dict[key]
except KeyError:
    value = default

# ✅ Good
value = my_dict.get(key, default)

```
### 7. Don't let low-level exceptions()bubble up raw to the top()
* low-level- exceptions -  exceptions that come from the deep technical internals of your program — things like databases, files, networks, and hardware
* top-  The top is the place where your program starts execution — the outermost layer
* raw- Raw means untranslated, unwrapped, and unprocessed — the exception travels from where it occurred (low level) all the way to the top exactly as it was, with no context added, no explanation, no meaningful message.eg : exception happened at low level but it becomes raw exception at UI level.
* Most apps have layers: Database → Service → UI

```py
# Layer 1 — Database (low level)
def fetch_user_from_db(user_id):
    result = db.query(f"SELECT * FROM users WHERE id={user_id}")
    return result

# Layer 2 — Service (business logic)
def get_user_profile(user_id):
    return fetch_user_from_db(user_id)

# Layer 3 — UI (top level)
def show_profile_page(user_id):
    return get_user_profile(user_id)


❌ Bad — exception bubbles up raw to the top

def fetch_user_from_db(user_id):
    result = db.query(...)   # raises OperationalError if DB is down
    return result

def get_user_profile(user_id):
    return fetch_user_from_db(user_id)   # does nothing, just passes it up

def show_profile_page(user_id):
    try:
        return get_user_profile(user_id)
    except OperationalError:             # caught here at the TOP
        print("Something went wrong")

The UI layer has no idea what OperationalError means in the context of showing a profile page. It can't do anything useful with it.

✅ Good — caught at the right layer

# Layer 1 — DB layer catches DB-specific error, wraps it meaningfully
def fetch_user_from_db(user_id):
    try:
        result = db.query(...)
    except OperationalError as e:
        raise DatabaseUnavailableError("DB is down") from e  # meaningful!
    return result

# Layer 2 — Service layer catches and handles missing user
def get_user_profile(user_id):
    try:
        return fetch_user_from_db(user_id)
    except DatabaseUnavailableError:
        return None   # service knows what to do — return empty profile

# Layer 3 — UI only deals with UI concerns
def show_profile_page(user_id):
    user = get_user_profile(user_id)
    if user is None:
        print("Profile unavailable, try again later")

```
Each layer handles only what it understands. The raw OperationalError never leaks to the UI 

Simple rule: Ask yourself — "Does this layer have enough information to do something useful with this error?" If yes, handle it here. If no, let the layer below handle it first.

Use logging.exception() inside an except block — it automatically captures the full traceback.


Catching bad data at the entry point is cleaner than letting errors propagate deep into logic.

## `contextlib.suppress` in Python

contextlib.suppress is a context manager that **silently ignores specified exceptions** — a cleaner alternative to a try/except block where you'd just pass on an error.
Basic Usage
```py
from contextlib import suppress

with suppress(FileNotFoundError):
    os.remove("file.txt")  # No error if file doesn't exist
```
 This is equivalent to:

```py
try:
    os.remove("file.txt")
except FileNotFoundError:
    pass 

```
### a- Suppressing Multiple Exceptions

Pass multiple exception types as arguments:

```py
with suppress(FileNotFoundError, PermissionError):
    os.remove("/protected/file.txt")

```

### Examples
* Safe delete a dict key without KeyError:
  ```py 
  with suppress(KeyError):
    del my_dict["optional_key"] 

  ```

*  Skipping already-existing directory creation:  
  ```py
  
    with suppress(FileExistsError):
    os.mkdir("new_folder")
    # Though os.makedirs(exist_ok=True) is better here

  ```

* Ignoring S`topIteration` manually:
```py

with suppress(StopIteration): # suppressing StopIteration Error
    value = next(my_iterator)

```

-> **StopIteration**

`my_iterator = iter([1, 2])`

`
next(my_iterator)  # 1
next(my_iterator)  # 2
next(my_iterator)  # 💥 StopIteration — tape is empty
`

If the iterator has another item, that item is assigned to value.
If the iterator is exhausted, next() raises StopIteration.

### When to use suppress vs try/except

|**Scenario**|**Prefer**|
|:---|:---|
|You truly don't care about the exception|suppress|
|You need to log, inspect, or react to the exception|try/except|
|Multiple statements, only one might fail|try/except (more precise) to that statement only|
|Clean one-liner "best effort" operation|suppress|

“Clean one-liner” means a small, simple operation where ignoring one specific exception is not a problem.

:::info
   **Execution stops at the exception** — code after the failing line ,inside the with block is skipped.
   ```py
   with suppress(ValueError):
      x = int("bad")   # raises → suppressed and going out of the with block
      print("skipped") # this never runs
                                                                

   ```

:::
