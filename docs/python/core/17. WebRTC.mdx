## 1. What is WebRTC?

**WebRTC** is a **free, open-source technology** that enables **real-time communication (audio, video, and data)** directly between browsers or devices.

**In simple terms:** It lets two users **talk, video call, or share data directly from their browsers** — without installing any plugins or third-party apps.

**Key ideas:**
- Works using **JavaScript APIs + HTML5**
- Enables **peer-to-peer (P2P) communication**
- Removes the need for third-party software like Skype (before WebRTC, browsers couldn't natively support audio/video streaming, so dedicated apps like Skype were required)

* Peer-to-peer (P2P) communication means two devices talk directly to each other, without sending their data through a central server in between.

> **Common misconception:** WebRTC is often thought to work *only* in browsers. While it was originally designed for the web, it has evolved into a **universal standard for real-time communication** used across almost every platform (mobile apps, IoT devices, desktop software, etc.).

## 2. Why WebRTC Was Created
 
**Before WebRTC:**
- Real-time communication required **plugins** (e.g., Flash) — a plugin was needed to run interactive content on web pages.
- Needed **complex, dedicated server setups** to relay audio/video.
**WebRTC solves this by:**
- Standardizing APIs through **W3C** and **IETF**
- Allowing **secure, browser-based communication** with no plugins

* "standardizing" though thses 2 organization, here just means these two organizations wrote down the official rules that browser makers and network engineers had to follow, so that WebRTC would work the same way everywhere


### Standards Bodies Involved
 
| Organization | Full Form | Role |
|---|---|---|
| **W3C** | World Wide Web Consortium | Standardizes the **browser API layer** — standardizes web technologies like HTML, CSS, browser behavior, and Web APIs |
| **IETF** | Internet Engineering Task Force | Standardizes the **network/protocol layer** — how the internet communicates underneath |
 
* The JavaScript API surface is a subset of "Web APIs".
* **IETF** has protocols like **SDP** (Session Description Protocol),**ICE**,**STUN/TURN** protocols,**DTLS/SRTP**.

## 3. Key APIs in WebRTC (Very Important)

The **W3C** published formal specifications defining exactly how JavaScript should be able to talk to a browser's media and networking features. This is what gave us the actual functions developers call, like:
 
WebRTC revolves around **3 core APIs**:
 
### 3.1 `getUserMedia()`
- **Purpose:** Captures media from the user's devices
- **Accesses:**
  - 🎥 Camera
  - 🎤 Microphone
- **Example use:** Capturing local video input for a call


### 3.2 `RTCPeerConnection`
- The **heart of WebRTC**
- Establishes the connection between two peers
- **Handles:**
  - Audio/video streaming
  - Network negotiation
  - Encryption
  
* Network negotiation is the process where two peers figure out the best way to actually reach each other over the network.

### 3.3 `RTCDataChannel`
- Used for sending **non-media data** between peers, such as:
  - Text messages
  - Files
  - Game data

## 4. How WebRTC Works (Step-by-Step)
 
### Step 1: Media Capture
- The process begins locally: the app calls `getUserMedia()` to request access to the camera and microphone.
- The browser prompts the user for permission.
- If granted, a **`MediaStream`** object is created, containing the audio and video tracks.
### Step 2: Signaling (The "Handshake")
Before any media can flow, the two peers must exchange information about their media capabilities (codecs, resolution, IP address, etc.). This exchange uses **SDP (Session Description Protocol)**.
 
- **The Offer:** Peer A creates an "Offer" (an SDP object) and sends it to Peer B through a **Signaling Server** — a middleman server typically using **WebSockets or HTTP**.
- **The Answer:** Peer B receives the offer, sets it as its "Remote Description," creates an "Answer" (its own SDP), and sends it back to Peer A.
Every peer maintains two SDP "slots" in memory:
- A **signaling server** is just a regular server (built and hosted by whoever makes the app — not provided by WebRTC itself) that acts as the middleman to introduce two peers to each other before they can connect directly.
 
| Slot | Meaning |
|---|---|
| **Local Description** | "This is what I am capable of" — my codecs, security keys, IP |
| **Remote Description** | "This is what the other peer is capable of" |
 
> The "Offer" and "Answer" are both instances of the SDP object.
 
**Summary:** Devices exchange IP addresses and media capabilities.
 
> ⚠️ **Important:** WebRTC does **NOT** define how signaling happens. A separate server (e.g., using WebSocket) is required to perform this step.

### Step 3: Network Discovery (ICE & NAT Traversal)
While signaling is in progress, the peers must find a way to connect their IP addresses directly. This is handled by **ICE (Interactive Connectivity Establishment)**.
 
- **NAT (Network Address Translation):** The technology that lets multiple devices (phone, laptop, smart fridge) share a single public IP address to access the internet.
- **ICE Candidates:** Each peer generates a list of potential IP addresses and ports where it can be reached.
- **STUN Server:** Tells a device behind a NAT what its public IP address is.
- **TURN Server:** If a direct connection isn't possible (e.g., strict corporate firewalls), a TURN server acts as a **relay**, passing data between the two peers.
> Firewalls act as the "security guards" of a network, controlling what traffic is allowed in and out.
 
#### STUN vs TURN
 
| Server | Role | Used When |
|---|---|---|
| **STUN** | Discovers the device's public IP/port | Tried **first**, for direct P2P connection |
| **TURN** | Relays media/data between peers | Used **only if** a direct connection fails |
 
> ⚠️ **Important:** WebRTC does **not** use STUN and TURN simultaneously for data transfer. It always **tries STUN first**, and falls back to **TURN only if needed**.
 
### Step 4: Connection Establishment
Once SDPs have been exchanged and a valid network path is found via ICE candidates:
- `RTCPeerConnection`  API establishes the link between peers.
- The browser performs a security handshake using **DTLS/SRTP protcols** to ensure all data is encrypted.
- The peers "poke holes" through their firewalls to allow **UDP** data packets to flow.
- DTLS — Datagram Transport Layer Security
  - Think of DTLS as TLS (the same encryption tech behind HTTPS) adapted for UDP instead of TCP.
- SRTP — Secure Real-time Transport Protocol
  - It's built on top of RTP (Real-time Transport Protocol — the standard for streaming audio/video over a network), adding encryption and authentication.
|Protocol|Secures|Used for|
|:---|:---|:---|
|DTLS|Key exchange + the data channel|Setting up exchanges of encryption keys; encrypting text/files/game data|
|SRTP|Audio/video media|Encrypting the actual call media|

**The problem it solves:**
* By default, most home/office routers and firewalls follow this rule:

* "Allow outgoing traffic. Block incoming traffic — unless it's a reply to something we already sent out."

* This is great for security, but it's a problem for P2P: if Peer B tries to send Peer A data out of nowhere, Peer A's firewall sees it as an unsolicited incoming connection and blocks it.
**How "poking a hole" works:**

1. Peer A sends a small outgoing packet toward Peer B (even just a tiny test packet, as part of the ICE connectivity checks).
2. Peer A's own firewall sees this as outgoing traffic it approved, so it temporarily opens up a "hole" — it remembers: "I sent something to Peer B's IP/port, so I'll allow replies back from that same IP/port."
3. Peer B does the same thing in the other direction at the same time — sending its own outgoing packet toward Peer A.
4. Now both firewalls have "holes" open for each other's traffic, because each one thinks it just made an outgoing connection and is only allowing the expected reply.
5. Once both holes are open, real audio/video/data can now flow freely through, in both directions — without either firewall blocking it.

### Step 5: Streaming Media and Data
- Media tracks (audio/video) now stream **directly** from one peer to the other.
- If a Data Channel was created, files or text messages can be sent over the same connection.
- WebRTC continuously monitors connection quality and can adjust video resolution or bitrate if the network becomes congested.
### Quick Recap
- **Signaling** → setup only (exchanging connection info)
- **Actual communication** → peer-to-peer (direct media/data flow)

## 5. Simple Example: Starting a Video Call - IMP
 
1. Browser asks for camera/microphone permission.
2. Connection info is exchanged with the other user (signaling).
3. Network discovery takes place (ICE/STUN/TURN).
4. A direct connection is established.
5. Video/audio starts streaming instantly.

## 7. Advantages and Disadvantages
 
| Advantages | Disadvantages |
|---|---|
| **Low latency** — direct P2P connection means faster communication | **Browser compatibility issues** — not all browsers fully support all features |
| **No plugins** — works natively inside the browser | **High bandwidth usage** — especially for video calls |
| **Secure** — encryption enabled by default | **Complex setup** — requires a signaling server (WebSocket/HTTP) and handling NAT traversal complexity |
| **Cost-efficient** — less server usage since media flows P2P | |

## 8. Important Technical Concepts (Glossary)
 
| Term | Definition |
|---|---|
| **Peer-to-Peer (P2P)** | Devices communicate directly with each other, without routing media through a central server |
| **Signaling** | The exchange of connection information (SDP offers/answers, ICE candidates) before communication begins |
| **ICE (Interactive Connectivity Establishment)** | The process of finding the best/most direct network path between peers |
| **Codecs** | Used to compress and decompress audio/video data (e.g., **VP8** for video, **Opus** for audio) |

## 9. Final Summary
 
WebRTC is:
- A **modern web standard**
- A technology that enables **real-time communication directly in browsers** (and increasingly, across other platforms)
- Built on:
  - JavaScript APIs (`getUserMedia`, `RTCPeerConnection`, `RTCDataChannel`)
  - Peer-to-peer networking (with signaling, ICE, STUN/TURN for connection setup)
  - Built-in encryption (DTLS/SRTP)
**WebRTC is the backbone of modern real-time web communication apps.**
